# ver.3 検討メモ

## 1. 位置づけ

`ver.3` は、`ver.2` で整理する発語記録と後退表現の基盤を前提に、  
**語の意味的なまとまり** や **言い換え・幼児語・家庭内表現の関係** を扱うための拡張フェーズとする。

`ver.2` では、まず

- 発語記録
- 後退表現
- 共通的な保存整理

の整備を優先し、辞書や概念統合は行わない。

## 2. ver.3 で扱う主題

### 2.1 語の表現と意味の分離

同じ意味を持つが、発語としては別のものを整理できるようにする。

例:

- `パパ`
- `おとうさん`
- `とーたん`

これらは発話形としては別だが、意味としては近い。

`ver.3` では、少なくとも次の 2 層を扱えるようにする。

- 実際に言ったことば（surface form）
- 意味上のまとまり（concept）

### 2.2 幼児語・家庭内語・通常語の関係

`ver.3` では、次のような関係を扱う可能性がある。

- 幼児語 → 通常語
- 家庭内の呼び方 → 一般語
- 言い間違い → 近い語
- 意味が近い別表現

例:

- `わんわん` ↔ `いぬ`
- `ブーブー` ↔ `くるま`
- `ママ` ↔ `おかあさん`
- `パパ` ↔ `おとうさん`

### 2.3 複数形・活用形・発音の揺れ

`ver.3` では、意味が同じ・近いというだけでなく、同じ語の表現バリエーションも扱えるようにする。

例:

- `car` / `cars`
- `go` / `going`
- `eat` / `eating`
- `パパ` / `ぱぱ`
- `いぬ` / `わんわん`

ここでは、単純な「同義語辞書」ではなく、

- 複数形
- 活用形
- 幼児語
- 家庭内語
- 発音の揺れ
- 近似表現

を含む **表現バリエーション** として考える。

### 2.4 表現バリエーションの可視化

同じ意味カテゴリの中で、どのように表現が増えていったかを見えるようにする。

例:

- `パパ` を言える
- `おとうさん` も言えるようになる
- 状況によって使い分けが見られる

これは単なる語彙数の増加とは別の発達的な意味を持つ可能性がある。

## 3. ver.3 で辞書を扱う理由

意味のまとまりや表現バリエーションを扱うには、ある程度の辞書または語彙マッピングが必要になる。

しかし、これは `ver.2` に入れるには重い。

理由:

- 幼児語は家庭差が大きい
- 言い間違いは個人差が大きい
- 多言語家庭ではさらに複雑になる
- 複数形や活用形も言語ごとに扱いが異なる
- 辞書を入力 UX に直結させると負担が急増する

そのため、辞書や概念提案は `ver.3` の主題として扱う。

## 4. ver.3 の基本方針

### 4.1 入力を重くしない

基本入力は引き続き「実際に言ったことば」だけにする。

ユーザーに毎回次を入力させない。

- concept
- 同義語グループ
- 標準語への対応
- 幼児語分類

これらは補助的に扱う。

### 4.2 辞書は小さく始める

最初から大規模辞書を作る前提にはしない。

初期候補:

- `ママ` / `おかあさん`
- `パパ` / `おとうさん`
- `わんわん` / `いぬ`
- `ブーブー` / `くるま`

このような、ごく小さい curated 辞書から始める。

必要であれば、将来は表現バリエーションとして次も視野に入れる。

- `car` / `cars`
- `dog` / `dogs`
- `go` / `going`
- `run` / `running`

### 4.3 提案は必須ではなく補助

辞書や concept は、自動確定ではなく補助提案に留める。

例:

- `おとうさん` 登録時に `パパ` と同じ意味として扱いますか？
- `わんわん` を `いぬ` グループにまとめますか？

ただし、`ver.3` の初期段階ではこの提案 UI すら持たず、  
まずは concept モデルの下地だけを持つ可能性もある。

## 5. 想定データモデル

### 5.1 最小案

```ts
interface SpeechPayload {
  surface: string;
  speechStatus: SpeechStatus;
  phraseSource?: string;
  conceptId?: string;
  lemma?: string;
}
```

ここで:

- `surface` = 実際の発話
- `conceptId` = 意味上のまとまり（任意）
- `lemma` = 基本形 / 代表形（任意）

とする。

### 5.2 concept の管理

最初は辞書そのものを複雑にせず、concept は次のような軽い管理でもよい。

```ts
interface LexicalConcept {
  id: string;
  canonicalLabel: string;
  locale: 'ja' | 'en' | 'zh-Hant';
}

interface LexicalVariant {
  conceptId: string;
  surface: string;
  lemma?: string;
  variantType?: 'baby_talk' | 'family_term' | 'standard' | 'approximation' | 'plural' | 'inflected';
}
```

### 5.3 3層モデル

`ver.3` では、少なくとも次の 3 層を区別して考える。

1. `surface`
   - 実際に発話された形
2. `lemma`
   - 基本形 / 代表形
3. `concept`
   - 意味上のまとまり

例:

- `surface: cars`
- `lemma: car`
- `concept: vehicle`

別の例:

- `surface: パパ`
- `lemma: パパ`
- `concept: father`

この区別があると、

- 同じ意味だが別語であるもの
- 同じ語の複数形や活用形
- 幼児語と通常語

を無理なく扱いやすい。

## 6. 表示の方向性

### 6.1 詳細画面

ある語に対して:

- 実際の発話
- 同じ意味に属する他の語
- 同じ lemma に属する語形
- 初出順

が見られるとよい。

### 6.2 サマリー

将来的には、次の 3 軸を分けて見られる可能性がある。

- 発話の種類数
- 基本形の種類数
- 意味カテゴリの種類数

### 6.3 タイムライン

タイムライン上は従来どおり発話形を主役にする。

concept は補助情報に留める。

## 7. ver.2 との境界

`ver.2` ではやらないこと:

- 大規模辞書導入
- 自動的な同義語推定
- 自動的な複数形 / 活用形の正規化
- 幼児語の自動正規化
- concept の必須入力

`ver.3` で検討すること:

- 小さな curated 辞書
- concept の導入
- 同義語・家庭内語の整理
- 複数形・活用形・表現揺れの整理
- 表現バリエーションの可視化

## 8. ver.3 着手前に必要な前提

`ver.3` に入る前に、`ver.2` 側で次が整っていることを前提にする。

1. 共通レコードモデルの方向性が決まっている
2. speech / communication / regression の記録構造が安定している
3. timeline と detail の表示基盤が整理されている
4. backup と migration の方針が安定している

## 9. この文書の役割

この文書は、辞書や concept を `ver.2` に持ち込まないための境界線として使う。

つまり:

- `ver.2` では record 基盤と観察体験を整える
- `ver.3` では語の意味的整理と表現バリエーションの可視化を扱う

という役割分担を明確にするためのメモである。
