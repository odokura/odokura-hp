# hatsu-go アプリ仕様書

## 1. 目的

`hatsu-go` は、子どもの発語を日々記録し、ことばの成長を振り返るためのモバイルアプリです。

この資料は、現行アプリの仕様を整理し、次のアプリ開発で再利用できる判断材料を残すことを目的とします。

対象読者:

- 次のアプリを設計する開発者
- 現行機能の全体像を把握したい保守担当
- 要件の棚卸しをしたい企画担当

## 2. プロダクトの基本方針

### 2.1 コア価値

- 発語を簡単に記録できる
- ことばの増え方を時系列で見返せる
- 家族内で共有しやすい
- 個人情報に配慮しながら外部共有できる

### 2.2 UX 方針

- 記録は最短操作で完了させる
- 分析機能は複雑にしすぎず、成長が直感的に分かる形にする
- 共有は軽く、過度に警戒させず、ただし配慮は残す
- 継続しやすいことを優先する

### 2.3 実装方針

- 画面状態は React ローカル state を基本とする
- 永続化は `storage/` に集約する
- 文言は i18n 経由で管理する
- UI での一時しのぎより、ロジック層や保存層で整合性を担保する

## 4. 主な機能

### 4.1 発語記録

登録できる情報:

- 単語
- 発語日
- 発語状態
- 任意メモ
- 任意動画
- 二語文以上の場合の元フレーズ

発語状態:

- `speech_only`
- `understood_only`
- `understood`
- `multi_word`

### 4.2 記録一覧と詳細

- 最近の記録をホームで確認できる
- 一覧画面で全記録を確認できる
- 単語詳細画面で単語単位の履歴を確認、編集、削除できる

### 4.3 成長グラフ

- 月ごとの積み上げグラフを表示する
- 状態別の累積語数を可視化する
- 生年月日がある場合は月齢表示、ない場合は経過月表示とする

### 4.4 共有画像

- 記録内容から共有用画像を再描画して生成する
- アプリ内プロフィール情報をそのままスクリーンショットせず、共有用レイアウトを別途構成する
- 初回共有時のみ軽い案内を表示する

共有画像で設定可能な表示項目:

- 名前
- アイコン
- グラフの月齢表示
- 先月から増えた単語数

共有画像の基本構造:

1. ヘッダー
2. ハイライト
3. 詳細グラフ
4. フッター

フッター表示:

- `Hatsu-go`

### 4.5 プロフィール設定

- 名前
- 写真
- 性別
- 生年月日
- 日付表示形式

### 4.6 多言語対応

対応ロケール:

- 日本語 `ja`
- 英語 `en`
- 繁体字 `zh-Hant`

### 4.7 バックアップと復元

- 手動バックアップ作成
- バックアップ共有
- バックアップファイルからの復元
- パスキーによる保護

### 4.8 課金

- Basic / Premium のプラン概念あり
- Premium で動画保存上限やバックアップ関連機能を拡張

## 6. データ仕様

### 6.1 発語記録

中心データ:

```ts
type SpeechStatus = 'speech_only' | 'understood_only' | 'understood' | 'multi_word';

interface SpeechRecord {
  id: string;
  word: string;
  recordedAt: string;
  videoUri: string | null;
  speechStatus: SpeechStatus;
  memo?: string;
  createdAt?: string;
  updatedAt?: string;
  phraseSource?: string;
}
```

補足:

- `recordedAt` は発語日
- `createdAt` / `updatedAt` は登録順・更新順管理用
- データ移行で欠損 timestamp を補完する仕組みを持つ

### 6.2 アプリ設定

主要設定:

```ts
interface ShareImageSettings {
  showName: boolean;
  showIcon: boolean;
  showMonthAge: boolean;
  showMonthlyDiff: boolean;
}

interface AppSettings {
  childName: string;
  birthDate: string | null;
  photoUri: string | null;
  gender: 'male' | 'female' | null;
  dateFormat: 'auto' | 'YYYY/MM/DD' | 'DD/MM/YYYY' | 'MM/DD/YYYY';
  shareImage: ShareImageSettings;
}
```

補足:

- `shareImage` は後から追加された設定であり、古い保存データでも deep merge により後方互換を保つ

## 7. 主要ロジック

### 7.1 記録保存

- AsyncStorage に保存
- 保存時に `createdAt` / `updatedAt` を付与
- 更新時は `createdAt` を維持し `updatedAt` のみ更新

### 7.2 データ移行

既存データの互換維持として:

- 旧 `speechStatus` 値の移行
- timestamp 欠損データの補完

### 7.3 グラフ集計

- 月単位で記録を集計
- 単語ごとに各月時点の最新状態を採用
- データのない月も埋めて表示する
- 表示スケールは最低バー数、最低 Y 軸最大値を持つ

### 7.4 日付表示

- `auto`
- `YYYY/MM/DD`
- `DD/MM/YYYY`
- `MM/DD/YYYY`

共有画像やバックアップ画面では、この表示ルールを利用する

## 8. 共有画像の考え方

### 8.1 目的

- SNS やメッセンジャーで共有しやすくする
- 記録の継続動機を高める
- 個人情報を調整可能にする

### 8.2 配慮方針

- 共有内容は設定で ON/OFF を切り替えられる
- 初回共有時のみ案内を出す
- 強い警告ではなく、軽い導線にする

### 8.3 レイアウト方針

- 情報の優先順位が上から下へ自然に流れること
- ヘッダー: 誰の / いつの画像か
- ハイライト: 今回の成果
- グラフ: 詳細
- フッター: ブランド

### 8.4 SNS 向け判断

- 過度な説明を入れない
- フッターは短く保つ
- 共有の邪魔になる重い確認 UI は避ける

## 9. 多言語化の方針

ルール:

- 文言は `t()` 経由
- 新規キーは `ja / en / zh-Hant` を同時更新
- 長い文言で崩れやすい箇所を重点確認

崩れやすい箇所:

- グラフ凡例
- 軸ラベル
- 共有画像ヘッダー
- ボタン行
- 空状態文言

## 12. 今回の実装で得られた知見

### 12.1 良かった点

- 共有画像をアプリ画面のスクリーンショットではなく専用 UI として持ったこと
- 保存層で後方互換を担保したこと
- i18n を3ロケールで揃える運用を徹底したこと
- 純粋関数と保存ロジックのテストを先に固めたこと

### 12.2 次回アプリでも再利用しやすい考え方

- UI より先にデータ責務を整理する
- 共有機能は「配慮」と「拡散しやすさ」の両立で設計する
- README と仕様資料の役割を分ける
- AI 向け運用メモと人間向けドキュメントを分離する

### 12.3 次回改善候補

- 実績解除 UI
- 共有画像の追加バリエーション
- 共有フローのより明確な locale 制御
- テスト対象の画面フロー拡張
