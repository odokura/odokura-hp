# 発語記録の編集（単語を編集可能にする）

## 背景

現行は[発語記録モデル（現行仕様）](./record-model.md)のとおり、編集画面で **`word`（と `phraseSource`）はロック表示で編集できない**（`app/modal.tsx` の lockBox）。日付・動画・発語状態・メモのみ編集可。

[単語の集約](./word-grouping/index.md)で表現と意味のずれ・二重登録・表記ゆれを扱うにあたり、登録済みの単語そのものを直せないと柔軟性に欠ける。言い直し・打ち間違いを反映できず、かえって二重登録を生む。

## 変更方針

**編集画面で `word` を編集可能にする。**

- `app/modal.tsx` 編集時の `word` ロックボックスを `TextInput` に変える（新規登録と同じ入力）。
- バリデーションは新規と同じ（空不可など、`lib/validate-record.ts`）。
- 保存は既存の `updateRecord`。`createdAt` 維持・`updatedAt` 更新は現行のまま。

## 編集の単位（決定）

- **対象はそのレコード 1 件の surface**（その日の観測 1 件）。単語詳細から開いた 1 レコードを編集する。
- 「同じ単語の全レコードを一括リネーム」は **MVP では行わない**（→ 未解決の論点）。
  - 理由: 編集はもともとレコード単位。1 件だけ直したいときに全件を巻き込むのは過剰。

## 集約との関係

- 編集で surface が変わると、グルーピングが変わる:
  - 別の既存グループと L0′ 正規化一致になれば、その日から同一グループに見える（→ [重複判定基準](./word-grouping/dedup-levels.md)）。
- **`conceptId` が付いているレコードは、surface を変えても `conceptId` を維持する**（明示的なまとめを編集で壊さない）。
- `lemma` も同様にユーザー指定があれば維持。surface 変更で機械候補を出し直してよいが、自動確定はしない。

## phraseSource の扱い

- 二語文由来レコードの `phraseSource` は、当面ロックのまま（編集対象は `word` のみ）。
- `phraseSource` も編集可能にするかは後続検討。

## 受け入れ条件

- 編集画面で `word` を変更して保存でき、ホーム・一覧・詳細に反映される。
- 変更後の `word` で正しくグルーピングされる（完全一致／L0′）。
- `conceptId` 付きレコードの surface を編集しても、concept のまとまりが外れない。
- 空の `word` は保存できない（現行バリデーション踏襲）。

## MVP 決定と後続

- **一括リネーム** → 後続。MVP は単件のみ（→ 「編集の単位」）。打ち間違いを一度で直したい需要は認識しつつ、後段で扱う。
- **`phraseSource` の編集** → 決定: MVP はロック維持（編集対象は `word` のみ）。
- **`multi_word` と `phraseSource` の不一致** → 決定: 分割語の `word` を編集しても `phraseSource`（元フレーズ）はそのまま履歴として残す。不一致は許容し、「元はこう言った」の記録として扱う。
