# 一覧の検索性・一覧性

## 背景・課題

登録が 100 件規模に増えると、**「何を登録したか」をリストから探すのが難しくなる**。

現行の一覧（`app/(tabs)/list.tsx`）は次のとおりで、規模が増えると破綻しやすい。

- まとめ単位は `word` 完全一致（`latestPerWord`）
- **ソートは 3 種のみ**（更新順 / 登録順 / あいうえお順）
- **検索ボックスがない**
- フラットな `FlatList` を上からスクロールするだけ

→ 「探す（検索性）」と「見渡す（一覧性）」の両面で改善する。

## 目標

- **検索性**: 探したい語に最短で届く。表記ゆれ（`ぱぱ`/`パパ`）でも見つかる。
- **一覧性**: 全体を見渡し、目的のあたりへ素早く飛べる。
- 既存のソート・タップ遷移の体験は壊さない。

## 方針

### 1. 検索ボックス（インクリメンタル）

- 一覧タブ上部に検索入力を置き、入力に応じて即時フィルタ。
- **照合対象**: surface（`word`）／ concept の `label` ／ グループに含まれる surface。
  - グループ横断で当たる（`おとうさん` で検索しても `パパ` グループが出る）。
  - **memo は照合対象に含めない**（ノイズを避け、語そのものの検索に絞る）。
- **正規化して照合**: クエリと対象の両方を [dedup の正規化](./word-grouping/dedup-levels.md)で寄せる。
  - `normalizeKey`（全角半角・大小・空白）で大小・幅差を吸収。
  - さらに `foldForSuggestion`（カナ折り・長音）で `ぱぱ` クエリが `パパ` に当たる。
  - 部分一致（含む）で拾う。

### 2. 一覧性（見渡し・ジャンプ）

- **あいうえお順のとき、見出し（セクション）を出す**: `あ行 / か行 …`（または頭文字）でグルーピングし、sticky ヘッダーにする。
- **索引レール（MVP に含める）**: 画面端に `あ か さ た … A〜Z #` のジャンプ索引を置き、タップでそのセクションへスクロール（連絡先アプリ風）。
- **件数表示**: ヘッダーに総件数、各セクションに件数。
- 更新順 / 登録順のときはセクションなし（時系列の流れを保つ）。索引レールはあいうえお順のときだけ表示する。

### 3. 集約との相乗

- [単語の集約](./word-grouping/index.md)で行数自体が減る（`パパ`＋`おとうさん`＝1 行）→ 一覧性が直接改善。
- 検索はグループ内の全 surface を対象にするので、まとめても「どの言い方でも見つかる」。

## 読み（yomi）の扱いと限界

- `SpeechRecord` / `LexicalConcept` に `yomi?: string` を持つ（→ [データモデル](./word-grouping/data-model.md)）。見出し名（単語）・まとめ名（concept label）の入力中、IME 変換前のキー入力（かな）から自動推定して保存する。入力欄は持つが自動入力されるため、通常の入力負担は増えない（手動編集も可能）。
- あいうえお見出し・索引レールは、次の優先順で読みを解決して振り分ける:
  1. 保存された `yomi`（グループの代表レコード or 所属 concept の `yomi`）があれば、それを使う。
  2. `yomi` が無く、見出しラベルの先頭文字がかな/ローマ字ならそれで振り分ける（従来どおり）。
  3. ラベルが漢字等で 1.・2. が不可なら、curated 辞書（`lib/aggregation-suggest.ts` の `CURATED_GROUPS`）からの読みヒントを使う（かな表記→漢字表記の順で並んでおり、漢字表記の直前のかな表記をその読みとして採用。例: `わんわん`/`いぬ`/`犬` の `犬`→`いぬ`）。
  4. 3. でも見つからなければ、同じグループ内の他の surface にかな書きのものがあればその先頭文字を使う。
  5. それでも読みが取れない語は、頭文字の字種で `A〜Z` / `#（その他）` バケツに入れる（完全な五十音整列はしない）。
- 自動採取の限界: 1 語の入力途中で複数回かな→漢字変換を挟む長い入力（複数単語の連続変換）では、最初の変換以降のかなを取り損ねることがある。フリック入力など composition を経由しない入力方式でも取れないことがある。本アプリの主用途（短い単語・まとめ名の単発入力）では十分機能する想定で、3.・4. のフォールバックが残るのはこのため。

## MVP スコープ

- 検索ボックス（正規化・部分一致・グループ横断、memo は対象外）
- あいうえおソート時のセクション見出し＋件数
- **索引レール**（あいうえお順のときのジャンプ）
- 読み（yomi）の自動採取＋curated 辞書・同グループ surface を使った読み解決（上記）

後続: 日付範囲などの絞り込み、curated 辞書に無い語の読み解決、検索ヒットのハイライト、`zh-Hant` 向けの拼音/部首ベース索引。

## 一覧性の MVP 決定

- **デフォルトソート** → 決定: 初期表示は**あいうえお順**にする（従来の `updated` から変更）。理由: 見出し（セクション）・索引レールは `aiueo` のときだけ表示されるため、初期値を `updated` のままにすると新しい一覧性向上の体験（セクション・索引レール）に多くの利用者が気づかない。あいうえお順を既定にすることで、追加した一覧性向上の仕組みを初回表示から提供する。
- **セクション粒度** → 決定: **行単位**（`あ / か / さ …`）でセクション分け。セクションの並び順（バケツ分け）はロケールに関わらず共通で、かな（あ→わ）→ `A〜Z` → `#`（漢字・その他）の順。
- **索引レール（タップ索引）の構成** → 決定: レール（画面端のジャンプ索引）はロケール別プリセットで構成を切り替える。バケツ分け自体（上記のセクション順）は変えず、レール側の見せ方だけを縮約する。
  - `ja`: `あ・か・さ・た・な・は・ま・や・ら・わ`（10 個・個別）→ `A`（`A`〜`Z` を統合した 1 枠、ラベル表示は「A」）→ `#`（1 個） = 12 枠。
  - `en` / `zh-Hant`: `A`〜`Z`（26 個・個別）→ `あ`（あ〜わを統合した 1 枠、ラベル表示は「あ」）→ `#`（1 個） = 28 枠。
  - 統合枠（`ABC` / `あ`）は、その範囲内で最初にデータがあるバケツへジャンプする。範囲内に該当データが1件もなければ非活性（淡色・タップ無効）。個別の枠も同様に該当データが無ければ非活性。
  - 背景: 主要文字種がロケールで偏るため（`ja` はかな中心、`en`/`zh-Hant` はアルファベット中心）、両方を常に26＋10個別表示すると一方が常時死に枠になり、レールの縦領域が固定のままタップ領域を圧迫する（旧仕様の37枠固定では1枠あたり約14ptまで縮小し、タップ操作性を損なっていた）。レールのみロケール別に縮約し、データの格納先（バケツ分け）自体は変えないことで、手動スクロール時の到達箇所に影響を与えない。
  - 既知の限界: `zh-Hant` の漢字ラベルは現状どのバケツにも一致せず `#` に集約される（拼音/部首ベースの索引は未対応）。これは本改修の対象外で、別途バックログとする。
