# 重複判定基準（「同じ」のレベル）

「二重登録」を一律に扱うと誤る。surface の近さを段階に分け、扱いを変える。

核心の方針:

- **明らかなものだけ自動でまとめる**（＝ L0′ 正規化一致。確認しない）。
- **明らか以外（L1 以上）は自動でまとめず、ユーザーに確認する**。粒度はユーザーの回答で決まる。
- どのレベルでも **破壊的マージはしない**（surface レコードは常に残る）。

## レベル定義

| レベル | 定義 | 例 | 重複か | 既定の扱い |
| --- | --- | --- | --- | --- |
| **L0 完全一致** | `word` が文字通り同じ | `パパ` / `パパ` | 重複ではない（別日の観測＝履歴） | 現行どおり 1 グループ |
| **L0′ 正規化一致** | `normalizeKey` が一致 | `Car` / `car`、`ＡＢＣ` / `abc`、`␣パパ␣` / `パパ` | 実質同一（入力ゆれ） | **黙って 1 グループ（確認しない）** |
| **L1 近似一致** | `foldForSuggestion` が一致（`normalizeKey` は不一致） | `ぱぱ` / `パパ`、`ぶーぶー` / `ブーブー` | 同じ可能性が高いが断定不可 | **登録時に任意提案**（自動マージしない） |
| **L2 同一語の語形** | `lemma` が同じ | `car` / `cars`、`go` / `going` | 別形（集約したい） | 同 lemma グループ（任意） |
| **L3 同義（別語）** | `concept` が同じ | `パパ` / `おとうさん`、`わんわん` / `いぬ` | 別語（集約したい） | 同 concept グループ（任意） |

要点: **L0 / L0′ は「同じ語」**＝確認なしでまとめる。**L1 以上は「別かもしれない」**＝自動マージせず、ユーザーに委ねる。

## 正規化の定義

### normalizeKey（黙ってまとめる＝L0′）

打ち間違い・入力環境レベルの揺れだけを吸収する。**意味を変えない変換のみ**。

```ts
function normalizeKey(s: string): string {
  return s
    .normalize('NFKC')                 // 全角/半角・互換文字を統一
    .replace(/[ 　]+/g, ' ')  // 連続空白を 1 つに
    .trim()
    .toLowerCase();                    // 英字の大文字小文字
}
```

適用するもの: 全角半角・互換文字、前後/連続の空白、英字の大小。これらは「同じ語の入力差」であり別語になることはまずない → **確認せずまとめてよい**。

### foldForSuggestion（提案候補＝L1）

さらに寄せるが、稀に別語・別ニュアンスになりうるので **提案に留める**。

```ts
function foldForSuggestion(s: string): string {
  return normalizeKey(s)
    .replace(/[ァ-ヶ]/g, (c) =>          // カタカナ → ひらがな（小書きは小書きのまま）
      String.fromCharCode(c.charCodeAt(0) - 0x60))
    .replace(/[ー—―−~〜]/g, 'ー'); // 長音・ダッシュ類の記号を統一
}
```

適用するもの: カタカナ⇄ひらがなの折りたたみ、長音/ダッシュ記号の種類の統一。これにより `ぱぱ` / `パパ`、`ぶーぶー` / `ブーブー` を拾う。ただしカナの選び方がニュアンス差を持つ場合があるため、**自動確定せず「同じことばですか？」と聞く**。

### 絶対に正規化しないもの（別語を壊さない）

- **濁点・半濁点**（`は` / `ば` / `ぱ`、`か` / `が`）
- **小書きカナの大小**（`きや` / `きゃ`、`つ` / `っ`、`や` / `ゃ`）。※カナ折りたたみでも小書きは小書きのまま保つ。
- **長音の有無そのもの**（`コーヒー` / `コーヒ` のように音が変わるもの）。※長音“記号の種類”の統一はするが、長音の有無は変えない。
- **語幹化・ステミング**（`走る` / `走った`）。これは正規化ではなく [データモデル](./data-model.md) の `lemma`（L2）の領域。

## 多言語のスコープ

- 正規化と L1 提案は **同じ表記体系の中だけ**で行う。`dog` / `DOG` は L0′、しかし `いぬ` / `dog` は L1 ではなく **concept（L3）** の話。
- 言語をまたぐ「同じ意味」は concept で扱い、正規化では混ぜない（→ [ロードマップ](./roadmap.md) の「多言語家庭」論点）。

## グルーピングと保存の関係

- **保存する surface は生のまま**（実際に入力した形）。正規化はマッチング用に都度計算し、保存はしない。
- 表示の代表ラベルをどれにするか（最新の生 surface か正規化形か）は [表示の方向性](./display.md) で扱う。
- 集約は `word` 単位。二語文の `phraseSource` は対象外。

## 判定フロー（まとめ）

1. `normalizeKey` 一致 → **黙って同一グループ**（L0 / L0′）。
2. それ以外で `foldForSuggestion` 一致 → **L1 提案**（はい / いいえ / あとで）。
3. `lemma` 一致 → L2、`concept` 一致 → L3（[集約 UX](./aggregation-ux.md) の手動付与／提案で束ねる）。
4. いずれも **自動の破壊的マージはしない**。surface レコードは常に残る。
