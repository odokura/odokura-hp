# ロードマップ・前提・未解決論点

## MVP（第 1 段階）の確定

方針: **lemma と concept の両方を最初から持つ**。**明らかなものだけ自動でまとめ、それ以外はユーザーに確認する**（粒度はユーザーが決める）。

### 自動マージと確認の線引き

- **自動でまとめる（確認なし）＝ 明らかなものだけ**: [L0′ 正規化一致](./dedup-levels.md)（NFKC・空白・大小文字）。`Car` / `car`、全角半角差など、別語になり得ない入力ゆれに限る。
- **ユーザーに確認（自動マージしない）＝ 明らか以外すべて**:
  - L1 表記ゆれ（`ぱぱ` / `パパ`、カナ折り・長音）
  - L2 lemma（`cars` → `car` などの語形）
  - L3 concept（`パパ` / `おとうさん`、curated 辞書一致）
  - → 「同じことば／同じ意味ですか？」と聞き、はい／いいえ／あとで。**粒度はユーザーの回答で決まる**。

### MVP に含めるもの

- **データ**: `SpeechRecord` に `conceptId?` と `lemma?` を追加（任意・後方互換）。concept は最小マスタ `LexicalConcept { id, label }` を新規ストアに持つ（→ [データモデル](./data-model.md)）。
- **グルーピング**: `conceptId` → `lemma` → `normalizeKey` の優先順でまとめる。
- **自動マージ**: L0′ のみ（黙って統合）。
- **確認ベースのまとめ**:
  - 登録時、既存に L1 / L2 / L3 の候補があれば任意提案を出す。
  - 後から[単語詳細](./aggregation-ux.md)で手動まとめ／解除、`lemma` の手動編集。
- **簡単な自動候補生成（提案の“元”を作るだけ。確定はユーザー）**:
  - 英語の単純な複数形・活用（`-s` / `-ing` など）から `lemma` 候補を機械生成し、確認に回す。
  - 小さな curated 辞書（`ママ` / `おかあさん` 等）から `concept` 候補を出し、確認に回す。

### MVP の受け入れ基準

- `Car` / `car` は確認なしで 1 グループ（L0′ 自動）。
- `ぱぱ` / `パパ`、`cars` / `car`、`パパ` / `おとうさん` は**提案が出て、ユーザーが「はい」で初めてまとまる**。
- 「いいえ」なら別単語のまま。まとめは後から解除できる（非破壊）。
- **二語文（`multi_word`）の分割語も提案チェックの対象**。「わんわん きた」の分割語 `わんわん` が既存の `いぬ` concept と一致すれば、単語登録と同じ提案を出す（→ [集約 UX](./aggregation-ux.md)）。
- **まとめを「はい」で確定する前に、見出し名（ラベル）を確認・編集できる**。Alert の文言だけでどちらの表記が残るかを決めさせない（登録時の提案／後からの手動集約で共通の UI）。ただし**既存 concept への参加時は見出し編集を省略**し、文言も「既に◯◯としてまとめられています」に変える（→ [集約 UX](./aggregation-ux.md)）。
- **同じ正規化表記（normalizeKey）のレコードは、いずれかが concept に参加したら全部フォールドインされ、一覧で同じ見出しが複数行に分裂しない**（→ [データモデル](./data-model.md)）。
- `conceptId` / `lemma` 未設定の既存レコードは、L0′ 統合を除き見え方が変わらない。
- マイグレーション不要（任意フィールド追加のみ・破壊的変更なし）。

## lemma の扱い

- `lemma` は主に**英語の活用・複数形**（`car` / `cars`、`go` / `going`）の集約で効く。
- 日本語は surface ≒ lemma が大半なので、多くのレコードで `lemma` は surface と同じになる（持っていても害はない）。
- MVP では `lemma` の付与は **手動編集＋簡単な機械候補（確認付き）** で行い、自動確定はしない。

## 後続段階

2. **辞書・ルールの拡張** — curated 辞書の拡充、英語活用ルールの強化、提案精度の向上。
3. **可視化** — 発話の種類数 / 基本形の種類数 / 意味カテゴリの種類数、表現バリエーションの推移。

## 着手前に整っているべき前提

1. 共通レコードモデルの方向性が決まっている
2. 発語・コミュニケーション・後退表現の記録構造が安定している
3. timeline と detail の表示基盤が整理されている
4. backup と migration の方針が安定している

## 残課題（MVP 方針を確定）

- **concept の粒度** → 決定済み。明らかなもの以外は確認するため、粒度はユーザーの回答で決まる。
- **「明らか」の閾値（L1 を自動にするか）** → 決定: **MVP では L1 を自動にしない**（確認のまま）。カナの選択がニュアンス差を持ちうるため黙って統合しない。自動は L0′ のみ。誤統合の実害が観測されたら再検討（要観察）。
- **多言語家庭** → 決定: MVP では **手動マージは言語に関係なく可能**（ユーザーが `いぬ`/`dog` を束ねたいなら束ねられる）。辞書・自動提案による言語またぎは後段。
- **集約 UI の配置** → 決定: MVP は単語詳細に置く。専用「まとめ」画面は規模が増えたら検討。
