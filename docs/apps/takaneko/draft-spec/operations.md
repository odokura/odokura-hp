# 運用と設定

## 実行環境

Takaneko は自宅の単一 PC で常時稼働させる。SQLite DB、設定、ログ、フィクスチャはローカルディスクへ
保存する。DB、収集データ、API 認証情報を GitHub に保存しない。GitHub には TypeScript のコードと
非機密の設定テンプレートだけを置く。

高可用性、別 PC への自動フェイルオーバー、bot によるクラウドバックアップは持たない。
Windows での常駐はタスクスケジューラの自動起動（ログオン時 + 失敗時再起動）とし、
スリープ無効化を運用手順に含める。

ローカル分析サイト（lab）は別プロセスとして任意起動で運用する。lab の停止・障害は bot の収集・保存に
影響せず、bot の停止・再開の判断に lab は関与しない。lab は bot の DB へ書き込まないため、
設定変更・キルスイッチは常に bot の CLI で行う（→ [ローカル分析サイト](./local-site.md)）。

## 起動と再開

起動時は、SQLite の整合性を確認し、前回未完了の収集・分析・投稿候補を読み込む。
情報源ごとの取得位置を確認して、取得可能な未取得範囲を追補する。回復できない範囲は欠損として記録し、
その範囲に依存する企業ファクトは投稿候補にしない。

追補で取得した開示は通常どおり正規化・分析するが、公開時刻から 24 時間（文書鮮度）を超えた候補は
stale として投稿を抑止する。企業ファクト自体は保存され、lab で閲覧できる（→ [配信](./distribution.md)）。

PC 停止、再起動、回線断の間は新しい取得を行わない。別環境で自動的に処理を継続してはならない。

## 停止制御

停止制御は全体、情報源、ContentSpec の単位で持つ。停止中は新しい投稿候補を作らず、live の投稿も行わない。
停止・再開には理由、時刻、対象、実行者を operation_event に記録する。運用操作（キルスイッチ、設定取り込み、
日次要約、リプレイ、訂正関係の確認、system 訂正の発行）は CLI で行う。

次の状態では、該当ルールを安全側に停止する。

- 同一情報源の取得が連続して失敗した（連続失敗回数の上限は設定。初期値は再試行条件の maxAttempts = 3。
  上限到達で該当情報源を停止し、解除は手動とする）。
- SQLite の保存または整合性確認に失敗した。
- ContentSpec 分析が鮮度・根拠箇所・必須属性・数値検証の条件を満たさない。
- live の投稿前検証（アカウント ID・protected・フォロワー数）が失敗、または確認できない
  （→ [配信](./distribution.md)。全体キルスイッチ）。

無人運転のフェイルクローズとして、正常に完了した収集・分析サイクルの時刻を記録する（watchdog）。
判定は取得スケジュールの予定と比較して行う。稼働時間帯内で、直近の予定サイクル時刻から 60 分を
超えて正常サイクルが記録されない場合に、全体キルスイッチを入れて新規処理を止める。夜間・休日など
スケジュール上サイクルが予定されない時間帯には発火しない。起動直後は、稼働時間帯内で起動から
60 分以内に初回サイクルが完了しない場合も発火する（初回サイクルのハングを検出するため）。
解除は運用者の手動操作だけとする。

## 監視とログ

最低限、情報源ごとの最終取得時刻、保存失敗数、根拠箇所抽出失敗数、ContentSpec・版ごとの
候補数・確定数・抑止数、投稿候補数、投稿試行数（published・failed・unknown 別）、DB 容量を記録する。

bot は pino の構造化 JSON ログをローカルファイル（logs/YYYY-MM-DD.jsonl）へ追記し、サイクルごとの
件数要約と日次集計を出力する。初期は外部通知を送らず、重大障害は該当ルールの停止とプロセス終了コードで
表す。外部通知は初期スコープ外とする。

## 手動退避

bot はバックアップ機能を提供しない。運用者は定期的に SQLite DB をクラウドストレージへ手動退避する。
退避時は bot を停止するか、SQLite の整合スナップショットを作成してからコピーする。実行中の DB ファイルだけを
そのままコピーしてはならない。lab の stats DB の退避は lab 側の手順とする（→ [ローカル分析サイト](./local-site.md)）。

## private 運用

- bot と lab は作者本人の自宅 PC だけで使う。lab は localhost にだけ bind し、外部公開しない。
- bot の外部送信は、作者本人が管理する非公開（protected）X アカウントへの投稿だけとする
  （→ [配信](./distribution.md)）。外部通知（メール・Slack 等）の認証情報は持たない。
- X API の認証情報（OAuth 1.0a のキー・トークン）は、Git 管理外のローカルファイルにだけ置く。
  SQLite の設定版（config_version）にも保存しない。認証エラー（401・403）は再試行せず、
  該当ルールを停止して operation_event に記録する。
- 実行モードは dry_run と live の二値とする。dry_run は外部送信を行わずローカル記録・表示だけを行う。
  live への切替条件と投稿前検証は[配信](./distribution.md)に従う。

## 設定管理

対象銘柄、情報源、ContentSpec、配信ルール、実行モード、キルスイッチは、TypeScript のコードではなく
設定として管理する。設定はローカルの JSON ファイルから読み込み、zod で検証後に SQLite の config_version へ
版として取り込む。実際に使う設定は SQLite に保存した版だけとし、過去の企業ファクト・投稿がどの設定で
行われたかを再現できるようにする。JSON の雛形は GitHub に置いてよいが、実運用の設定と認証情報は置かない。

| 設定 | 初期方式 | 内容 |
| --- | --- | --- |
| 対象銘柄集合 | 手動コード一覧 | 銘柄コード、集合名、有効期間 |
| 情報源カタログ・取得スケジュール | 版付き JSON | 情報源、コネクタ、取得間隔・時間帯、タイムアウト、取得ペーシング、再試行条件 |
| 文書正規化マップ | 版付き JSON | 情報源の物理項目（XBRL 要素・コンテキスト等）と論理文書種別・source_excerpt の対応 |
| ContentSpec | 版付き JSON | 種別、必須項目、抽出方式、根拠規則、テンプレート、優先度、重複項目 |
| 配信ルール | 版付き JSON | 頻度・選抜の上限、鮮度、文書内の選抜、表示順 |
| 実行モード | 版付き JSON | dry_run / live。live は投稿先アカウント ID（数値）と月間予算上限を必須項目とし、欠けた設定は取り込みを拒否する |
| キルスイッチ | SQLite の操作記録 | 全体・情報源・分析の停止状態 |

X API の認証情報は設定版に含めず、Git 管理外のローカルファイルで別管理する（→ private 運用）。

設定ファイルの実例:

~~~json
// app.json（初期）
{ "mode": "dry_run", "timezone": "Asia/Tokyo" }

// app.json（live の例。dry_run ゲート通過後にだけ取り込む → 配信）
{
  "mode": "live",
  "timezone": "Asia/Tokyo",
  "x": { "accountId": "1234567890", "monthlyBudgetUsd": 5 }
}

// universe.json
{ "name": "core", "codes": ["7203", "6758", "8306"] }

// sources.json
{
  "tdnet": {
    "schedule": { "days": "weekdays", "from": "07:30", "to": "19:00", "intervalMinutes": 10 },
    "fetch": {
      "timeoutSeconds": 30,
      "minFetchIntervalSeconds": 2,
      "maxDocumentsPerCycle": 30,
      "maxListPagesPerCycle": 20
    },
    "retry": { "maxAttempts": 3, "backoffMinutes": [1, 5, 15] }
  }
}

// distribution-rules.json
{
  "perDocument": 3,
  "perInstrumentPerDay": 6,
  "perKindPerDay": 8,
  "perAccount": { "per15Min": 5, "perDay": 10 },
  "minIntervalSeconds": 60,
  "freshnessHours": 24,
  "window": { "from": "00:00", "to": "24:00" }
}
~~~

### ContentSpec の取り込み検証

ContentSpec を取り込む前に、次を検証する。

1. 種別が[共通契約](./content/common.md)の ContentKind に一致する。
2. acceptedDocumentKinds、extractionMode、requiredFields、evidenceRule、renderTemplate、priority、dedupeFields を持つ。
3. structured-parser は、必要な要素・行・列・単位・期間の対応を文書正規化マップで解決できる。
4. text-rule と llm-candidate は、許可見出し、許可行為、必須主語・対象・時点、返却 JSON スキーマを持つ。
5. テンプレートは必須表示項目を含み、未検証の属性や自由文を参照しない。
6. 訂正文書用の固定表示項目、文字数上限、種別固有の重複項目を定義する。

文書タイトルだけを投稿可能にする設定、根拠のない自由要約を出力する設定、必要な検証を持たない設定は無効とする。

### 検証と適用

設定を取り込む前に、JSON 形式、銘柄コード、数値範囲、時間帯、ルール間の矛盾を検証する。
無効な設定は現在有効な設定を置き換えず、理由を operation_event に残す。

有効な設定は新しい版として保存し、次の分析・投稿サイクルから適用する。すでに作成済みの company_fact は
作成時の版を保持し、新しい設定で書き換えない。

## 初期設定値

全初期値の正本。変更は設定版の追加で行い、コード変更では行わない。

| 設定 | 初期値 |
| --- | --- |
| 投稿対象種別 | 財務 3 種（決算実績・業績予想の修正・配当）。ほかは凍結（→ [スコープと段階計画](./scope.md)） |
| 対象銘柄集合 | 手動コード一覧、30 銘柄以下 |
| 取得スケジュール | TDnet を平日 07:30〜19:00 JST に 10 分間隔 |
| 取得の負荷抑制 | HTTP タイムアウト 30 秒、原本取得の間隔 2 秒以上、1 サイクルの原本取得 30 件・一覧再走査 20 ページまで（超過分は次サイクルへ持ち越し、持ち越しを記録） |
| 情報源の停止閾値 | 同一情報源の連続取得失敗 3 回で該当情報源を停止（手動解除） |
| 文書鮮度 | 公開時刻から 24 時間以内。超過は stale 抑止（追補分を含む） |
| 文書あたりの投稿数 | 最大 3 件。priority と元文書の出現順で選ぶ |
| 投稿頻度 | 1 暦日 10 件、15 分 5 件、最小間隔 60 秒（→ 上記 distribution-rules.json） |
| 投稿本文の上限 | 重み付き長 280（→ [共通契約](./content/common.md)） |
| watchdog | 稼働時間帯内で、直近の予定サイクルから 60 分正常サイクルがない場合に全体停止。初回サイクルは起動から 60 分以内の完了を要求 |
| 実行モード | dry_run。live への切替は dry_run ゲート通過後に設定版の追加で行う（→ [配信](./distribution.md)） |
| X 予算上限 | 月 5 USD。Developer Console 側の予算上限も同額に設定する |

初期値は、作者本人が検証済みの一次情報を非公開アカウントと localhost で確認するための保守的な値である。
