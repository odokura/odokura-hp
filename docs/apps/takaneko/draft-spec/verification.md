# 検証計画

## 方針

Takaneko は作者本人だけが使う private アプリケーションであり、外部送信は作者管理の非公開（protected）
X アカウントへの投稿に限る（→ [配信](./distribution.md)）。一次情報の収集、保存、正規化、企業ファクト、
投稿候補、投稿記録を再現可能な固定データで検証する。第三者への再配信・外部通知は実装も検証も行わない。

実在する情報源への接続試験は、作者が利用可否を手動確認し、運用上問題ないと判断した後だけに行う。
X への実投稿は、下記の dry_run ゲートを通過した後だけに行う。

## 固定データ

有効な種別ごとに、正常文書、必須項目欠落、曖昧な要素、数値不整合、訂正文書、重複文書を網羅する。
フィクスチャは許可された情報源から採取し、原本を含むため Git 管理外の `fixtures/` に置く。期待値は
作者本人が原本と突合して確定する。

実開示の採取が M0 の利用条件確認にブロックされる間は、手作りの合成データで実装とテストを進めてよい。
ただし M1 以降の完了判定と「要確認」項目の確定には、実開示から採取したフィクスチャだけを使う。

## 単体テスト

| 対象 | 確認すること |
| --- | --- |
| 文書判定 | 表題パターン、メタデータ抽出、XBRL 添付有無 |
| 文書正規化 | 見出し階層、段落位置、XBRL 要素・コンテキスト・単位を source_excerpt として再特定できる |
| structured-parser | 要素候補の一意解決、あいまいな要素・単位・期間の拒否 |
| 財務比較 | 実績値の抽出、前年同期比較、会計期間・単位・連結範囲の取り違え防止 |
| 投稿文 | 固定テンプレートが企業名・コード・分類・時点・根拠を含み、重み付き長 280 以内であること |
| 重複 | 同一文書・根拠箇所・主題の再処理で company_fact を重複作成しないこと |
| 設定 | 無効な ContentSpec・設定 JSON を拒否すること。既定テンプレートの mode が dry_run であること。投稿先アカウント ID・予算上限を欠く live 設定を拒否すること |
| 投稿前検証 | アカウント ID 不一致、protected でない、フォロワー数が 0 でない、確認失敗の各ケースで投稿を中止し全体キルスイッチが入ること（X API はモック） |

## 結合テスト

一時 SQLite DB と固定の一次情報文書を使い、保存、正規化、根拠箇所、ContentSpec 選択、企業ファクト、
投稿候補までを通して確認する。

1. 表形式の決算文書から、検証可能な一つの数値比較を含む company_fact を作る。
2. 公式タイトルだけの文書・XBRL なしの文書は保存するが、投稿候補を作らない。
3. 同じ企業ファクトの再処理が重複レコードを作らない。
4. 訂正文書が原記録を上書きせず、訂正関係を保存する。
5. 保存失敗、キルスイッチ、根拠不足が新しい投稿候補を抑止する。
6. DB 再起動後も収集位置・設定版・企業ファクトを保持する。
7. dry_run で post_attempt に status = dry_run が記録され、外部送信が発生しない。

## リプレイ検証

フィクスチャ一式を SQLite に投入し、ContentSpec と対象銘柄集合の指定版で再生する CLI を用意する。
文書数、種別別の company_fact 数、投稿候補数、抑止理由別・銘柄別の件数を出力し、同じ入力と
設定版から同じ結果になることを確認する。

## 運用検証

- SQLite マイグレーション途中の失敗で bot が起動しない。
- 保存失敗と DB 整合性エラーで新規分析・投稿が停止する。
- 正常サイクルの途絶（初回サイクルの未完了を含む）で watchdog が全体キルスイッチを入れる。
- lab の読み取り専用接続を開いたまま bot の保存と WAL チェックポイントが継続する。
- bot / lab が localhost 外で待ち受けない。lab は外部送信を行わない。bot の外部送信は
  live モードの X API 呼び出し（投稿・自アカウント確認・結果照合）だけである。dry_run では一切送信しない。
- GitHub に DB、収集データ、フィクスチャ原本、認証情報が含まれない。

## dry_run ゲート（live 切替の条件）

live モードへの切替は、次を全て満たした後だけに行う。判定結果と根拠は operation_event に記録する。

1. 許可された実データを対象に、14 日間以上 dry_run で常駐運転している。
2. その間に生成された投稿候補が、有効な種別ごとに 5 件以上ある。閑散期で不足する種別は、
   実開示フィクスチャのリプレイで補完してよい（補完した事実を記録する）。
3. 投稿候補の数値・単位・会計期間・根拠・本文に誤りがないことを、作者本人が全件確認した。
4. リプレイ検証で同じ入力・設定版から同じ結果が再現される。
5. 投稿前検証（アカウント ID・protected・フォロワー数）の単体テストが通過し、
   live 設定（アカウント ID・予算上限）と Developer Console の予算上限を設定済みである。

ゲート通過後も、live の初週は投稿件数を日次で確認し、誤投稿時は[配信](./distribution.md)の
手順で対応する。

## 実装完了の基準

TypeScript の型検査、種別別の単体テスト、結合テスト、固定データによるリプレイ検証、Docusaurus ビルドが
成功し、生成される投稿文がコンテンツ仕様を満たすことを確認する。さらに、上記 dry_run ゲートの
1〜4 を満たすことを作者本人が確認する（live への切替自体は完了基準に含めない。切替は運用判断とする）。
