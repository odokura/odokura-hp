# 検証計画

## 方針

投稿先が本番 X アカウントで手動承認を持たないため、種別ごとの抽出、判定、投稿、停止制御は
実装前後に再現可能なデータで検証する。種別ごとの具体的なフィクスチャ一覧は
[コンテンツ仕様](./content/common.md)の各種別ページを正本とし、本ページは共通のテスト構成と
完了基準を定める。

実在する情報源への接続試験は M0 の利用条件確認後に行う。ここでは、情報源に依存しない
判定・保存・配信の検証を定める。

## 固定データの分類

有効な種別ごとに、少なくとも次の分類を網羅するフィクスチャを作る（具体的なケースは各種別ページ）。

| 分類 | 確認すること |
| --- | --- |
| 正常文書 | 必須項目と根拠箇所を持つ company_fact と固定テンプレートの投稿候補を作る |
| 必須項目欠落 | 欠けた属性を推測せず、抑止理由を残す |
| 曖昧・類似表現 | 要素・見出し・許可行為が一意でない候補を抑止する |
| 数値不整合 | 単位、会計期間、連結範囲、比較元が違う数値を比較しない |
| 文字数超過 | 定型置換後も上限を超える本文を抑止する |
| 訂正文書 | 原記録を上書きせず、訂正表示を持つ新しい候補を作る |
| 重複文書 | 重複キーにより再投稿を防ぐ |

フィクスチャは実開示から採取し、原本を含むため git 管理外の fixtures/ に置く。期待値（本文・件数・
抑止理由）は人手で確定し、数値表示は開示 PDF の印字と一致させる。

## 単体テスト（vitest）

| 対象 | 確認すること |
| --- | --- |
| 文書判定 | 表題パターン、メタデータ抽出、XBRL 添付有無の判定 |
| 文書正規化 | 見出し階層、段落位置、XBRL 要素・コンテキスト・単位を source_excerpt として再特定できる |
| structured-parser | 要素候補の一意解決、あいまいな要素・単位・期間の拒否 |
| text-rule / llm-candidate（Phase 2 以降） | 許可見出し・許可行為・引用の一意出現検証、スキーマ外出力の拒否 |
| 財務比較 | 実績値の抽出、前年同期比較、会計期間・単位・連結範囲の取り違え防止 |
| 投稿文 | 固定テンプレートだけを使い、銘柄・分類・一つの具体的事実を含め、重み付き文字数を守る |
| 重複・頻度 | 同一企業ファクト、文書上限、銘柄上限、分類上限、アカウント上限、最小間隔、時間帯外を抑止する |
| 設定 | 無効な ContentSpec・設定 JSON を拒否し、設定版を過去の候補へ遡及適用しない |

## 結合テスト

一時 SQLite DB と固定の一次情報文書を使い、保存、正規化、根拠箇所、ContentSpec 選択、企業ファクト、
投稿候補、X アダプタの模擬応答までを通して確認する。実 X API は呼ばない。

次の経路を必須とする（MVP では財務 3 種の該当分。文章系の経路は有効化時に追加する）。

1. 表形式の決算文書から、検証可能な一つの数値比較を含む投稿候補を作る。
2. 公式タイトルだけの文書・XBRL なしの文書は保存するが、投稿候補を作らない。
3. 一文書の四つ目の候補が、文書あたり上限により抑止される。
4. 同じ企業ファクトの再処理が重複投稿を作らない。
5. 別文書に載った同一事実（同じ重複キー）が再投稿されない。
6. 訂正文書の投稿候補が、訂正表示を本文に含み、同一重複キーで投稿できる。
7. unknown の X 投稿結果が自動再送されない。
8. キルスイッチ、鮮度切れ、頻度超過、保存失敗が投稿を抑止する。
9. DB 再起動後も、投稿済み・送信中・不確定の状態を保持する。
10. 業績予想と配当の複合開示が、種別間で二重投稿を作らない。
11. dry_run モードで送信が発生せず、post_attempt に dry_run として記録される。
12. 決算集中日相当の多候補入力で、一括選抜が承認済み未送信・送信中・不確定の候補を頻度枠に含めて数え、
    アカウント・銘柄・分類の上限を超えない。

## リプレイ検証

フィクスチャ一式（一次情報文書、source_excerpt、financial_fact）を SQLite に投入し、
ContentSpec・対象銘柄集合・配信ルールの指定版で再生する CLI（scripts/replay.ts）を用意する。
再生結果として、文書数、種別別の候補数、company_fact 数、投稿候補数、抑止理由別・銘柄別の件数を
出力する。同じ入力と各設定版から同じ結果になることを確認する。

ContentSpec、優先度、対象銘柄集合、頻度上限を変更した場合は、同じデータセットで再生し、
変更前後の企業ファクト数と投稿件数の差を確認してから有効化する。

## 運用検証

- SQLite マイグレーションの途中失敗で bot が起動しない。
- 保存失敗と DB 整合性エラーで新規分析と投稿が停止する。
- プロセス停止後に、未完了の候補と設定版を保持して再開する。
- 正常サイクルの途絶（6 時間）で watchdog が全体キルスイッチを入れる。
- DB の停止後コピーまたは整合スナップショットを別の SQLite で開ける。
- 読み取り専用接続（ローカル分析サイト相当）を開いたまま、bot の保存・投稿・WAL チェックポイントが継続する。
- 訂正前の文書・根拠箇所・ContentSpec・企業ファクト・投稿監査記録を上書きせず保持する。
- GitHub に DB、収集データ、フィクスチャ原本、認証情報が含まれない。

## 実装完了の基準（live 切替ゲート）

TypeScript の型検査、各種別の単体テスト、結合テスト、固定データによるリプレイ検証、
Docusaurus ビルドがすべて成功し、初期設定で生成される投稿がコンテンツ仕様を満たすことを確認する。

さらに、実データを対象にした dry_run モードで 14 日間以上運転する。期間に加えて、種別ごとの
最低サンプル数を満たすまでゲートを通過しない（初期値）: 決算実績の正常 company_fact 10 件・
抑止 5 件、配当 5 件、業績予想の修正 3 件、訂正ケース 1 件、重複抑止 1 件。決算閑散期などで
実開示が不足する分は、保存済みの実開示データをリプレイで再生して補完してよい。

生成された全投稿本文について、銘柄・数値・単位・会計期間の誤りと重複がないことを確認し、
抑止理由の分布に想定外の大量抑止がないことを見る。dry_run 中に誤った本文が 1 件でも見つかった場合は、
原因の ContentSpec を修正し、期間を仕切り直す。

これらをすべて満たしてから本番 X アカウントを有効化する（→ [スコープと段階計画](./scope.md) M3・M4）。
