# Takaneko 設計判断と未決事項

確定した設計判断と、その理由を残す。仕様本体は各ページを正とし、ここは判断の要点をまとめる。

## 設計判断

| 判断 | 理由 |
| --- | --- |
| 速報通知ではなく、企業理解の事実を届ける | 速報は既存サービスと重複する。評価語なしの事実蓄積が差別化であり、プロダクトの約束そのもの |
| 一投稿一企業ファクト。評価語・因果補完・自動要約を禁止 | 中立性と検証可能性の土台。数値と比較基準の併記で判断材料は足りる |
| MVP は財務 3 種（決算実績・予想修正・配当）に絞る | XBRL で正しさを機械検証できる唯一の領域。文章系は検証機構なしでは誤投稿リスクが工数に見合わない |
| OCR・PDF 表解析を行わず、XBRL 添付がない開示は投稿しない | 表構造の安定取得が成立するのは XBRL のみ。安全側に倒し PDF 解析を完全回避する |
| 情報源は TDnet 第一候補、EDINET は Phase 3 | 短信は EDINET に載らず、四半期報告書廃止後は四半期数値の適時取得が短信のみ |
| 投稿重複キーは意味属性のみ（版・ハッシュ・文書 ID を含めない） | 設定の版上げ・再正規化・別文書経由の同一事実で再投稿しないため。訂正だけ明示バイパス |
| 決算実績は実績値と比較差を一つのファクトに統合 | ファクト粒度とテンプレート（一投稿）を一対一にするため |
| 配当の重複キーに金額を含める | 同額再掲は抑止し、修正（金額変化）は新キーで投稿するため |
| LLM 候補は原文の完全一致引用を返す（オフセット禁止） | LLM はオフセット計算を安定して行えない。引用の一意出現を検証器が確認して span 化する |
| dry_run モードを持つ（承認フローではない） | 手動承認なし・検証用アカウントなしの構成で、本番投稿前に品質を測る唯一の手段 |
| watchdog による全体フェイルクローズ（正常サイクル 6 時間途絶で停止） | 外部通知なしの無人運転で、異常な収集・分析が続くことを防ぐ |
| アカウント頻度 1 暦日 10 件で開始 | 従量課金下でタイムラインの読みやすさと費用（月 5 USD 未満）を抑える保守初期値。Developer Console の予算上限で費用を天井する |
| 認証は OAuth 1.0a ユーザーコンテキスト | リフレッシュトークンの失効・更新運用がなく、無人常駐と相性がよい |
| 根拠 URL を投稿に表示しない | 外部リンク付き投稿はリーチが下がる傾向があり、従量課金の単価も高い。検証可能性は内部来歴と再現可能な値で担保する |
| 追記型の正本・無期限保存・SQLite 単一ファイル | 投稿の来歴再現（信頼性）に必須。単一 PC 運用で扱える規模 |
| 単一 PC・手動退避・高可用性なし | bot の停止はフォロワーへの実害が小さく、誤投稿リスクのほうが重い |
| 対象銘柄は手動一覧・30 銘柄以下で開始 | 頻度上限・鮮度 24 時間と整合させ、決算集中日の大量抑止を限定する |
| 決算集中日の抑止（翌日持ち越しなし）を仕様として受け入れる | 古い決算を流し続けるより当日の事実を優先。公平化は価値の実測後 |
| ORM・cron ライブラリ・キュー・Web フレームワークを使わない | 単一プロセス直列で足りる。依存を増やさない |
| ContentSpec の版を過去の company_fact・投稿へ遡及適用しない | 過去の投稿がどの設定で行われたかを再現するため |
| bot は TypeScript を維持し、Python へ移行しない | 短信サマリーの iXBRL は平坦で自前パーサが最小。難所の指標対応表はライブラリでも自前になる。二言語化は常駐運用のコスト増 |
| Python の導入は正規化 CLI の差し替えと lab に限定 | 信頼性核（bot）と分析（lab）で要求が異なる。M1 で自前パーサが破綻したときだけ、同一フィクスチャで検証できる CLI 境界で切り替える |
| bot と lab は作者本人専用の private アプリケーションとする | 外部投稿・第三者再配信・外部通知を持たず、SQLite と localhost の表示だけに限定する。X API と live 切替は現行スコープから除外する |
| セクター動向分析はローカル分析サイト（lab）で行い、X には配信しない | 多社集計は一投稿一企業ファクトの契約外。非公開表示なら統計データの再配信条件も問われない |
| lab と bot はプロセス・DB 書き込み権・障害ドメインを分離 | 表示・分析の障害や負荷が投稿の信頼性へ波及しないようにする。bot DB の書き手は bot のみ |
| 統計データは lab の stats DB（別ファイル）に保存 | bot 正本 DB の単一ライター原則と監査範囲を守る。統計は再取得で復元可能 |

## 2026-07 仕様レビューの反映

初版仕様（9 種別を同粒度で詳細化した構成）への批判的レビューで見つけた矛盾・欠落と、その反映。

| 指摘 | 反映 |
| --- | --- |
| 投稿重複キーに ContentSpec 版・配信ルール版・内容ハッシュが含まれ、版上げのたびに再投稿できてしまう | 意味属性のみのキーへ変更、訂正バイパスを追加 |
| 配当の重複キーに金額がなく、配当修正が重複として抑止される | キーに金額を追加 |
| 決算実績の実績値と前年同期差が別ファクトなのにテンプレは一投稿併記 | 一ファクトに統合し baseline_type で区別 |
| LLM に UTF-16 オフセットを返させる契約は満たせない | 完全一致引用契約へ変更 |
| 文字数上限が未定義 | 重み付き長 280 と定義 |
| 頻度制御にバースト・API 上限の考慮がない | 最小間隔 60 秒、1 暦日 10 件、銘柄・分類別上限を追加 |
| 本番投稿前に品質を測る手段が仕様上ない | dry_run モードと 14 日間ゲートを追加 |
| 異常時に気付く手段がない | watchdog フェイルクローズを追加 |
| X 自動化ルールへの言及がない | 自動化ラベル・運営者明示を運用要件に追加 |
| 情報源未選定のまま抽出詳細が確定していた | TDnet 第一候補と採用前チェックリストを定め、XBRL 要素は「要確認」としてフィクスチャで確定する手順に変更 |
| 9 種別を同時に詳細化しており、文章系の実現可能性が未検証 | 財務 3 種を MVP とし、文章系は凍結してゲートを設定 |

構成もこのとき再編した: 論理仕様と物理仕様の二層を種別ページへ一体化し、共通契約・スコープ・
決定ログへ重複を単一化した。

## 2026-07 追加検討（Python・セクター分析・ローカルサイト）

三つの追加要望を検討した結果。

| 論点 | 結論 |
| --- | --- |
| Python の XBRL ライブラリを使うために部分・全体移行すべきか | 移行しない。MVP の対象（短信サマリー iXBRL）は範囲が狭く自前パーサが最小で、ライブラリが効く EDINET フルタクソノミは Phase 3。導入境界は正規化 CLI の差し替え（M1 の保険）と lab に限定（→ [アーキテクチャ](./architecture.md)の言語方針） |
| セクター（複数企業）と官庁・工業会の統計データによる動向分析 | lab の機能として仕様化。セクターは手動定義の銘柄集合、統計は遡及改定を考慮して取得時点ごとに追記保存。X へは配信しない（→ [ローカル分析サイト](./local-site.md)） |
| 収集結果・分析サマリーのローカルサイトと X bot の分離運用 | 二実行体に分割。lab は bot DB を読み取り専用（WAL）で参照し、独自の stats DB を持つ。localhost のみ・外部公開なし。最初のユースケースは dry_run レビュー画面（→ [ローカル分析サイト](./local-site.md)） |

## 2026-07 外部レビュー（Codex）反映

実装前レビュー（verdict: block、P1 8 件・P2 1 件）の指摘と反映。全件を妥当と判断して反映した。

| 指摘 | 反映 |
| --- | --- |
| watchdog が夜間・休日に必ず発火する（6 時間固定） | 取得スケジュールの予定サイクル基準に変更（稼働時間帯内で予定から 60 分。非稼働時間帯は判定しない） |
| X API の前提が旧プラン制（無料 500 件・読み取り不可） | 従量課金（2026-02 以降の既定）へ更新。M0 で Console 確認を必須化、予算上限を初期値に追加、unknown 照合を条件付きで許可 |
| decimals を表示単位として使用、テンプレートで符号が二重 | 表示単位は scale と単位ヘッダから決定し decimals は精度検証に限定。表示値は原文文字列、差分は符号なし絶対値 + 符号欄に統一 |
| 再取得テストが source_record の UNIQUE と矛盾、カーソルが行位置依存 | 取得試行を source_poll に分離。カーソルは公開時刻・開示番号基準で直近 24 時間を重複再走査 |
| dry_run ゲートが期間のみで、閑散期は件数ゼロで通過できる | 種別ごとの最低サンプル数を追加。不足分は実開示のリプレイで補完 |
| 承認済み未送信候補が頻度枠に数えられず上限超過し得る | 一括選抜で approved・sending・unknown を枠として予約し、送信直前に再判定 |
| bot の誤解析を訂正する経路がない | correction_kind = source / system を導入。system 訂正は運用者が CLI で対象投稿を指定して発行 |
| 配当キーが現在額のみで、再修正の戻りが抑止される。文書内選抜も想定とずれる | 配当を初出（金額なしキー）と変更（修正前後の額をキーに含む）に分離。文書内サブ上限 1 件を明示 |
| 設定版が整数のみで外部キーがない（P2） | post_candidate の対象集合・配信ルール版を config_version(id) への外部キーに変更 |

## 未決事項

| 論点 | 次に決めること |
| --- | --- |
| TDnet の利用条件 | M0 のチェックリストで機械取得・作者本人のローカル保存・分析の可否を確認する。無料閲覧サイトは robots.txt により不採用であり、契約API・書面許可・代替情報源を決める |
| XBRL の物理仕様 | 要素名・コンテキスト命名・パッケージ構成・予想修正と剰余金配当への添付有無を、フィクスチャ採取（M0〜M1）で確定する |
| 対象銘柄集合の中身 | 運用者が 30 銘柄以下で決める。決算月の分散を考慮 |
| 予想修正の「未定」の扱い | 前回未定・取り下げ・レンジ予想は初期は抑止。dry_run の抑止統計を見てテンプレート追加を判断 |
| 決算集中日の公平化・有償プラン | live 運用の抑止統計とフォロワー反応を見て判断 |
| 会社予想比の baseline | 保存済みの予想 financial_fact との突合を Phase 2 で設計 |
| 増減要因・修正理由ファクト | Phase 2 の text-rule と同時に company_explanation として設計 |
| PDF・HTML 文章の位置表現と訂正識別 | 文章系種別（Phase 2〜3）の有効化時に決める |
| Python 正規化 CLI への切替 | M1 のフィクスチャで TypeScript 自前パーサの成否を見て判断（トリップワイヤ） |
| 統計情報源の選定と利用条件 | e-Stat・工業会統計から採用系列を選び、プログラム取得・保存の可否を確認する（lab L1 の前提） |
| セクター定義の粒度と管理 | 手動マッピングの単位（東証 33 業種準拠か独自か）を L1 着手時に決める |
| lab の Web 実装方式 | 軽量サーバーか静的生成かを L0 着手時に決める。localhost のみ・読み取り専用は確定 |
| セクター分析の X 配信 | 当面行わない。行う場合は新種別として設計指針の 6 項目とゲートを満たす |
