# 収集と保存

## 情報源の扱い

情報源は、採用前に保存・手動クラウド退避・X 配信の可否を確認済みのものだけを登録する。bot は実行時に
ライセンス判定をせず、登録済みのコネクタだけを動かす。

MVP の情報源は TDnet のみとする。採用理由とチェックリストは[スコープと段階計画](./scope.md)に定める。
企業ごとの公式サイトを個別にクロールするアダプタは、将来種別の有効化まで追加しない。
リアルタイム株価、出来高、指数などの市場データは扱わない。官庁・工業会の統計データは bot の
情報源ではなく、ローカル分析サイトが独自の DB へ収集する（→ [ローカル分析サイト](./local-site.md)）。

情報源ごとの取得頻度、レート制限、障害時の再試行条件は情報源カタログ（設定）で定義する
（→ [運用と設定](./operations.md)）。

## TDnet からの取得

本節の取得方式は、情報源の利用条件確認（M0）が完了するまでの暫定仕様である。2026-07 時点で
無料の閲覧サイトは robots.txt によりプログラム取得不可と判断しており、契約 API・書面許可・
代替情報源のいずれになるかで一覧の構造・カーソルの実体は変わり得る（→ [スコープと段階計画](./scope.md)）。
決定まで実情報源へ接続しない。以下の論理設計（差分取得・カーソル・負荷抑制）は経路によらず維持する。

- 平日 07:30〜19:00 JST に 10 分間隔で開示一覧を取得する（値は初期設定。→ [運用と設定](./operations.md)）。
- 一覧の各行（時刻、コード、会社名、表題、PDF・XBRL への参照）を読み、対象銘柄の未保存の開示だけ
  原本（XBRL zip と PDF）を取得する。対象外銘柄の開示は取得も保存もしない。一覧のコード表記
  （5 桁等）と銘柄マスターのコードの照合規則は、実データで確定する（要確認）。
- カーソルは開示の公開時刻と開示番号（外部 ID）を基準にする。一覧の日付・ページ・行位置には
  依存しない（後からの挿入やページ境界の変化で開示を飛ばすため）。毎サイクル、カーソルから
  直近 24 時間分を重複再走査し、外部 ID で保存済みをスキップする。
- 取得試行は、新規開示の有無にかかわらず source_poll として記録する（実行時刻、結果、カーソル、
  新規保存件数）。再開時はこのカーソルから追補する。
- アクセス間隔・時間帯は、利用条件の確認結果（M0）で見直す。

### 負荷抑制

システムが担う負荷抑制は技術的なものに限る（低頻度アクセス、対象銘柄の限定、営業時間帯の限定、
再試行上限、タイムアウト）。値は情報源カタログの設定で持ち、初期値は[運用と設定](./operations.md)に定める。

- すべての HTTP 要求にタイムアウトを設定する。
- 原本取得（zip・PDF）は要求間に最小間隔を置き、1 サイクルの取得件数に上限を設ける。
  上限を超えた分は次サイクルへ持ち越し、持ち越しの発生を source_poll に記録する。
- 一覧の重複再走査は 1 サイクルに読むページ数へ上限を設け、超過分は次サイクルへ持ち越す。
- 決算集中日でも上限は緩めない。取りこぼしは持ち越しで追補する。

### 再試行

- 同一サイクル内では再試行しない。取得に失敗したサイクルは失敗として source_poll に記録し、
  次サイクル以降の再試行とする。連続失敗時の待機は backoffMinutes に従い、通常間隔より長く空ける。
- 連続失敗回数が maxAttempts（初期 3）に達したら、該当情報源のキルスイッチを入れて取得を停止する。
  解除は運用者の手動操作だけとする（→ [運用と設定](./operations.md)）。
- 連続失敗回数は source_poll の記録から導出し、別の状態を持たない。

## 収集フロー

~~~text
情報源コネクタ
      │
      v
未加工レスポンスの検証
      │
      v
source_record へ追記保存
      │
      v
文書・表・文章の正規化
      │
      v
source_excerpt と論理文書種別を保存
      │
      v
ContentSpec による分析要求を作成
~~~

原本保存が成功するまで、正規化・分析・投稿へ進めない。保存に失敗した場合は、取得失敗として記録し、
同じ情報源を無制限に再試行しない。

## コネクタの契約

各情報源コネクタは、未加工データのほかに、情報源 ID、外部 ID、公開時刻、取得時刻、参照先、
更新後のカーソルを返す。呼び出し側は取得試行を source_poll に記録し、原本は source_record に
一意保存する（同一内容の再取得では source_record を増やさない）。

コネクタは情報源固有の形式を sources の外へ出さない。正規化失敗や対象外データは原本と失敗理由を残し、
投稿対象にしない。

## 正規化と根拠箇所

文書正規化マップは、情報源の物理的な表現を ContentSpec が使う論理文書種別と source_excerpt へ変換する。

- HTML・テキスト文書は、見出し階層、段落番号、文の文字範囲、内容ハッシュを保持する。
- 表形式（XBRL サマリーを含む）は、要素・コンテキストまたは表名・見出し行・行・列、セル値、
  単位・注記、内容ハッシュを保持する。
- 明示メタデータは、情報源が付与した文書種別、公開時刻、企業コードなどを保持する。
- 画像だけの PDF や構造を確定できない表は原本を保存するが、structured-parser の入力にしない。

銘柄、公開時刻、論理文書種別が不明な文書は、保存しても分析・投稿には使わない。会計期間が不明な文書は、
会計期間を必要とする数値比較に使わない。訂正前後の関係を判断できない文書も、投稿対象の分析を抑止する。

## 重複と訂正

同じ原本を再取得した事実は source_poll に残り、source_record は同一内容を重複保存しない。
正規化済みデータとして重複する場合は、同一の ContentSpec 分析を繰り返さない。情報源が訂正した場合は、訂正原本と訂正前原本の関係を保存し、新しい正規化データと
企業ファクトを作る（種別ごとの訂正規則は各コンテンツ仕様ページ）。

## 単一 PC での保存

SQLite は bot のローカルディスクに置き、ネットワークファイルシステム上で直接開かない。bot は自動バックアップや
クラウド転送を行わない。運用者が DB を手動で退避する際は、bot 停止後または SQLite の整合スナップショットを使う。

容量不足、DB 整合性エラー、保存失敗が起きた場合は、新しいデータに基づく分析と投稿を停止する。
