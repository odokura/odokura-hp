# スコープと段階計画

[設計指針](./design-principles.md)の価値判断に基づき、いま作る範囲と、範囲を広げる条件を定める。
判断の経緯は[決定ログ](./decisions.md)に記録する。

Takaneko は作者本人だけが自宅 PC で使う private アプリケーションである。外部投稿・第三者再配信・
外部通知は本スコープに含めない。

## 合意した提供価値（MVP）

MVP で有効化するのは次の 3 種別だけとする。いずれも「数字と資金の使い方はどうか」の問いに答える。

| 種別 | MVP に入れる根拠 |
| --- | --- |
| 決算実績（financial_result） | 決算短信の XBRL サマリーから機械可読で取得でき、値の検証が完結する |
| 業績予想の修正（financial_forecast_revision） | 修正開示の XBRL から前回・今回を同一条件で取得できる |
| 配当（shareholder_return の配当部分） | 短信サマリー・配当予想修正の XBRL から取得できる |

3 種別とも抽出は structured-parser のみ。text-rule と llm-candidate は使わない。OCR・PDF の表解析は
一切行わず、XBRL が添付されない開示は原本保存のみとしてローカル表示候補にしない。

この絞り込みは価値の放棄ではない。数値ファクトは 9 種別の中で唯一、正しさを機械検証できる。文章系の価値は、検証機構がないまま
並行実装せず、数値系で運用の信頼を確立してから積む。

## 実現可能性の評価（9 種別）

抽出方式と情報源の機械可読性による評価。価値の有無ではなく、誤ったローカル分析を検証可能な費用で
抑えられるかの評価である。

| 種別 | 抽出の実体 | 機械可読な情報源 | 判定 |
| --- | --- | --- | --- |
| 決算実績 | 表（structured-parser） | 決算短信の XBRL サマリー | MVP |
| 業績予想の修正 | 表（structured-parser） | 予想修正開示の XBRL（添付範囲は要確認） | MVP |
| 配当 | 表（structured-parser） | 短信サマリー・配当予想修正の XBRL | MVP |
| 自己株式取得 | PDF 本文の文章 | なし（XBRL 非対象） | Phase 2 |
| 資金調達・買収・売却・設備投資 | PDF 本文の文章 | なし | Phase 2 |
| 事業構造の変化 | 文章 + LLM 候補 | なし | Phase 3 |
| 戦略・数値目標・資源配分 | 文章 + LLM 候補 | なし | Phase 3 |
| 事業の実行・進捗 | 文章 + LLM 候補 | なし | Phase 3 |
| リスク・制約 | 文章 + LLM 候補 | なし | Phase 3 |
| ガバナンス・経営体制 | 文章（対象役職で限定） | なし | Phase 3 |

将来種別の論理仕様は[将来種別（凍結）](./content/future.md)に維持する。

## 段階計画とゲート

| Phase | 範囲 | 有効化ゲート |
| --- | --- | --- |
| 0 | 収集・保存・正規化の骨組み + private ローカル表示基盤 | フィクスチャ全通過、ライセンス上許可された実開示が SQLite に貯まる |
| 1 | 財務 3 種の private 分析・localhost 表示 | [検証計画](./verification.md)の完了基準（private 運転の期間・種別別最低件数ゲートを含む） |
| 2 | 自己株式取得、資金調達・買収・売却・設備投資（text-rule） | 対象開示 30 文書以上の固定データで適合率 100%、取りこぼしの理由が全件説明可能 |
| 3 | 文章系 5 種（text-rule + llm-candidate）、EDINET・企業サイトの追加 | 種別ごとに Phase 2 と同じ基準。llm-candidate は引用検証の棄却率も計測 |

ゲートを満たさない種別は、期限を理由に有効化しない。Phase 2 以降の着手は、Phase 1 の運用が
1 か月安定する（抽出誤りゼロ、watchdog 発火なし）まで凍結する。

## マイルストーン

| # | 内容 | 完了条件 |
| --- | --- | --- |
| M0 | 情報源の利用条件確認（下記チェックリスト）、フィクスチャ採取、DDL・設定取り込み・常駐ループ・一覧差分取得 | 許可された実開示の source_record が貯まり、再起動でカーソルから追補される |
| M1 | 短信 XBRL の正規化と決算実績の確定ロジック | [決算実績](./content/financial-result.md)のフィクスチャ全通過。「要確認」の要素・コンテキストが確定 |
| M2 | 予想修正・配当の確定ロジック、ローカル表示テンプレート展開 | 全種別のフィクスチャ通過。リプレイで再現一致 |
| M3 | private 常駐運転（14 日以上） | [検証計画](./verification.md)の private 運転ゲート通過 |

## 情報源の方針

### TDnet を第一候補とする理由

- 決算短信・業績予想修正・配当予想修正は適時開示であり、TDnet に集約される。EDINET（法定開示）には短信が載らない。
- 2024 年に金商法の四半期報告書が廃止され、四半期数値の適時取得は短信が事実上唯一の経路になった。
- 対象 3 種別の開示には XBRL サマリーが添付される（対象範囲と形式はフィクスチャ採取で確定する）。

EDINET API は、有価証券報告書・半期報告書が必要になる Phase 3 まで採用しない。
取得間隔・差分検出などの取得方式は[収集と保存](./collection.md)に定める。

### 採用前チェックリスト（M0 の前提）

情報源の採用は[設計指針](./design-principles.md)のとおり人手で確認してから登録する。TDnet では次を確認する。

1. 閲覧サービスの利用条件が、プログラムによる定期取得を禁じていないか。
2. 取得したデータの保存と手動クラウド退避が利用条件に反しないか。
3. 取得したデータを作者本人のローカル DB に保存・分析することが利用条件に反しないか。
4. robots.txt と、アクセス間隔・負荷に関する記載。
5. 一覧・文書 URL の安定性（外部 ID として使えるか）。

いずれかが不可なら、有償の開示データフィードの費用対効果を評価するか、MVP を再設計する。
この確認より先に実装を進めない。

## 対象銘柄集合

- 手動のコード一覧、30 銘柄以下で開始する。作者本人が確認可能な件数に絞るためである。
- 選定は作者本人の裁量とする。
- 銘柄マスターは、この一覧に対応するコード・企業名・市場区分を手で用意すれば足りる。
  全上場銘柄マスターの自動取込は作らない。

## 併走コンポーネント: ローカル分析サイト（lab）

収集結果・抽出・抑止・ローカル表示候補のサマリーと、官庁・工業会の統計データを使ったセクター動向の分析は、
bot と分離した非公開のローカル分析サイトが担う（→ [ローカル分析サイト](./local-site.md)）。

- lab は bot の Phase・マイルストーンから独立に進め、bot のゲート条件にしない。
- ローカル表示画面（L0）は M3 の確認作業を効率化するため、M2 完了後の早期着手を推奨する。
- セクター分析の結果は作者本人の localhost 閲覧だけに使う。

## スコープ外（bot の MVP）

- 文章系 5 種別と自己株式取得・資本取引のローカル表示（論理仕様は[将来種別（凍結）](./content/future.md)に維持）
- 増減要因・修正理由ファクト（会社明示の説明。Phase 2 の text-rule と同時に設計）
- LLM の一切の利用
- EDINET・企業公式サイトのアダプタ
- 全上場銘柄マスターの自動取込
- X API、投稿、投稿済みポストの削除・返信スレッド
- 外部通知（メール・Slack 等）
- セクター動向の外部配信（分析自体は lab で行う。→ [ローカル分析サイト](./local-site.md)）
- 官庁・工業会の統計データの bot による収集（lab が独自 DB へ収集する）
