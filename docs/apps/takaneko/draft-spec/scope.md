# スコープと段階計画

[設計指針](./design-principles.md)の価値判断に基づき、いま作る範囲と、範囲を広げる条件を定める。
判断の経緯は[決定ログ](./decisions.md)に記録する。

## 合意した提供価値（MVP）

MVP で有効化するのは次の 3 種別だけとする。いずれも「数字と資金の使い方はどうか」の問いに答える。

| 種別 | MVP に入れる根拠 |
| --- | --- |
| 決算実績（financial_result） | 決算短信の XBRL サマリーから機械可読で取得でき、値の検証が完結する |
| 業績予想の修正（financial_forecast_revision） | 修正開示の XBRL から前回・今回を同一条件で取得できる |
| 配当（shareholder_return の配当部分） | 短信サマリー・配当予想修正の XBRL から取得できる |

3 種別とも抽出は structured-parser のみ。text-rule と llm-candidate は使わない。OCR・PDF の表解析は
一切行わず、XBRL が添付されない開示は原本保存のみとして投稿しない。

この絞り込みは価値の放棄ではない。数値ファクトは 9 種別の中で唯一、正しさを機械検証でき、誤投稿が
フォロワーの信頼を壊すリスクを工数に見合う形で抑えられる。文章系の価値は、検証機構がないまま
並行実装せず、数値系で運用の信頼を確立してから積む。

## 実現可能性の評価（9 種別）

抽出方式と情報源の機械可読性による評価。価値の有無ではなく、誤投稿リスクを検証可能な費用で
抑えられるかの評価である。

| 種別 | 抽出の実体 | 機械可読な情報源 | 判定 |
| --- | --- | --- | --- |
| 決算実績 | 表（structured-parser） | 決算短信の XBRL サマリー | MVP |
| 業績予想の修正 | 表（structured-parser） | 予想修正開示の XBRL（添付範囲は要確認） | MVP |
| 配当 | 表（structured-parser） | 短信サマリー・配当予想修正の XBRL | MVP |
| 自己株式取得 | PDF 本文の文章 | なし（XBRL 非対象） | Phase 2 |
| 資金調達・買収・売却・設備投資 | PDF 本文の文章 | なし | Phase 2 |
| 事業構造の変化 | 文章 + LLM 候補 | なし | Phase 3 |
| 戦略・数値目標・資源配分 | 文章 + LLM 候補 | なし | Phase 3 |
| 事業の実行・進捗 | 文章 + LLM 候補 | なし | Phase 3 |
| リスク・制約 | 文章 + LLM 候補 | なし | Phase 3 |
| ガバナンス・経営体制 | 文章（対象役職で限定） | なし | Phase 3 |

将来種別の論理仕様は[将来種別（凍結）](./content/future.md)に維持する。

## 段階計画とゲート

| Phase | 範囲 | 有効化ゲート |
| --- | --- | --- |
| 0 | 収集・保存・正規化の骨組み + dry_run 基盤（投稿なし） | フィクスチャ全通過、実開示が SQLite に貯まる |
| 1 | 財務 3 種の live 投稿 | [検証計画](./verification.md)の完了基準（dry_run の期間・種別別最低件数ゲートを含む） |
| 2 | 自己株式取得、資金調達・買収・売却・設備投資（text-rule） | 対象開示 30 文書以上の固定データで適合率 100%、取りこぼしの理由が全件説明可能 |
| 3 | 文章系 5 種（text-rule + llm-candidate）、EDINET・企業サイトの追加 | 種別ごとに Phase 2 と同じ基準。llm-candidate は引用検証の棄却率も計測 |

ゲートを満たさない種別は、期限を理由に有効化しない。Phase 2 以降の着手は、Phase 1 の運用が
1 か月安定する（誤投稿ゼロ、watchdog 発火なし）まで凍結する。

## マイルストーン

| # | 内容 | 完了条件 |
| --- | --- | --- |
| M0 | TDnet 利用条件の確認（下記チェックリスト）、X Developer Console での単価・権限・予算上限・照合 API の確認、フィクスチャ採取、DDL・設定取り込み・常駐ループ・一覧差分取得 | 実開示の source_record が貯まり、再起動でカーソルから追補される。頻度初期値と unknown 照合方針を確定 |
| M1 | 短信 XBRL の正規化と決算実績の確定ロジック | [決算実績](./content/financial-result.md)のフィクスチャ全通過。「要確認」の要素・コンテキストが確定 |
| M2 | 予想修正・配当の確定ロジック、配信判定、テンプレート展開 | 全種別のフィクスチャ通過。リプレイで再現一致 |
| M3 | dry_run 常駐運転（14 日以上） | [検証計画](./verification.md)の dry_run ゲート通過 |
| M4 | X アダプタ疎通と live 切替 | 自動化ラベル設定済み、live 初週の全投稿を日次ログで人手確認し誤りゼロ |

## 情報源の方針

### TDnet を第一候補とする理由

- 決算短信・業績予想修正・配当予想修正は適時開示であり、TDnet に集約される。EDINET（法定開示）には短信が載らない。
- 2024 年に金商法の四半期報告書が廃止され、四半期数値の適時取得は短信が事実上唯一の経路になった。
- 対象 3 種別の開示には XBRL サマリーが添付される（対象範囲と形式はフィクスチャ採取で確定する）。

EDINET API は、有価証券報告書・半期報告書が必要になる Phase 3 まで採用しない。
取得間隔・差分検出などの取得方式は[収集と保存](./collection.md)に定める。

### 採用前チェックリスト（M0 の前提）

情報源の採用は[設計指針](./design-principles.md)のとおり人手で確認してから登録する。TDnet では次を確認する。

1. 閲覧サービスの利用条件が、プログラムによる定期取得を禁じていないか。
2. 取得したデータの保存と手動クラウド退避が利用条件に反しないか。
3. 開示から抽出した事実を X で配信することが利用条件に反しないか。
4. robots.txt と、アクセス間隔・負荷に関する記載。
5. 一覧・文書 URL の安定性（外部 ID として使えるか）。

いずれかが不可なら、有償の開示データフィードの費用対効果を評価するか、MVP を再設計する。
この確認より先に実装を進めない。

## X アカウント運用の制約

- **料金モデル**: X API は従量課金である（2026-02 以降の新規開発者の既定。公表単価の例: 投稿 1 件
  0.015 USD、リンク付き投稿は 0.20 USD、読み取り 1 件 0.005 USD 前後）。単価・権限は変動し得るため
  Developer Console を正とし、M0 で確認する。1 暦日 10 件（月最大約 310 件）は月 5 USD 未満に収まる
  見込み。Console の予算上限を設定し、超過による送信ブロックは配信ルール停止として扱う
  （値の正本は[運用と設定](./operations.md)の初期設定値）。
- **読み取り**: 定期的な読み取りは行わない。投稿結果が不確定（unknown）になった場合の照合だけを、
  M0 で確認した照合手段と費用の範囲で行う（→ [配信](./distribution.md)）。
- **リンクなし投稿**: 根拠 URL を表示しない方針（→ [決定ログ](./decisions.md)）は、リンク付き投稿の
  高単価により費用面でも支持される。
- **認証**: OAuth 1.0a のユーザーコンテキストを使う。リフレッシュトークンの失効・更新運用が存在せず、
  無人常駐と相性がよい。
- **自動化ルール**: 自動化アカウントラベルと運営者アカウントの明示を運用要件とする（→ [運用と設定](./operations.md)）。
- **プロフィール**: 投資判断への免責、bot であること、扱う情報の範囲（一次情報の事実のみ）を記載する。

## 対象銘柄集合

- 手動のコード一覧、30 銘柄以下で開始する。頻度上限と文書鮮度 24 時間のもとで、決算集中日の
  大量抑止を限定するためである。
- 選定は運用者の裁量とするが、決算月が分散していると集中日の抑止が減る。
- 銘柄マスターは、この一覧に対応するコード・企業名・市場区分を手で用意すれば足りる。
  全上場銘柄マスターの自動取込は作らない。

## 決算集中日の挙動（合意事項）

対象 30 銘柄中 8 社が同日 15 時台に短信を開示した場合の想定動作。

- 各社の短信から最大 3 件（配当 1 + 営業利益 1 + 売上高 1 が典型。配当は文書内 1 件のサブ上限
  → [配当](./content/dividend.md)）で最大 24 候補が出る。
- アカウント上限 1 暦日 10 件・15 分 5 件・最小間隔 60 秒により、優先度・出現順で 10 件だけが
  当日中に投稿され、残りは抑止理由付きで保存される。
- 抑止された候補は翌日に持ち越さない。文書鮮度 24 時間を過ぎた候補は投稿しない。
  古い決算を翌日以降に流し続けるより、当日の事実だけを届けることを優先する。

集中日の取りこぼしが実運用で価値を損なうと分かった場合に、銘柄間の公平化・上限の引き上げ・
有償プランを検討する（→ [決定ログ](./decisions.md)の未決事項）。

## 併走コンポーネント: ローカル分析サイト（lab）

収集結果・抽出・抑止・投稿のサマリー表示と、官庁・工業会の統計データを使ったセクター動向の分析は、
bot と分離した非公開のローカル分析サイトが担う（→ [ローカル分析サイト](./local-site.md)）。

- lab は bot の Phase・マイルストーンから独立に進め、bot のゲート条件にしない。
- dry_run レビュー画面（L0）は M3 の確認作業を効率化するため、M2 完了後の早期着手を推奨する。
- セクター分析の結果は X へ投稿しない。配信する場合は新種別として
  [設計指針](./design-principles.md)の 6 項目とゲートを満たす別判断とする。

## スコープ外（bot の MVP）

- 文章系 5 種別と自己株式取得・資本取引の投稿（論理仕様は[将来種別（凍結）](./content/future.md)に維持）
- 増減要因・修正理由ファクト（会社明示の説明。Phase 2 の text-rule と同時に設計）
- LLM の一切の利用
- EDINET・企業公式サイトのアダプタ
- 全上場銘柄マスターの自動取込
- 投稿済みポストの削除・返信スレッド
- 外部通知（メール・Slack 等）
- セクター動向の X 配信（分析自体は lab で行う。→ [ローカル分析サイト](./local-site.md)）
- 官庁・工業会の統計データの bot による収集（lab が独自 DB へ収集する）
