# 将来種別（凍結）

> 現行の Takaneko は作者本人だけが使う private アプリケーションであり、X 投稿は作者管理の
> 非公開（protected）アカウントに限る（→ [配信](../distribution.md)）。本ページの種別は凍結中であり、
> 将来の抽出論理を保存するだけで、投稿テンプレート等の記述を有効な機能仕様として扱わない。

自己株式取得と、Phase 2・3 の種別の論理仕様。機械可読な情報源がなく、text-rule または llm-candidate の
検証データがそろっていないため、[スコープと段階計画](../scope.md)のゲートを通過するまで有効化しない。
本ページの仕様は価値定義と抽出条件の論理レベルであり、有効化時に種別ページへ昇格させ、
情報源への物理束縛・抑止理由コード・フィクスチャを追加する。

各種別の優先度と重複キー属性は末尾の表にまとめる。

## 自己株式取得（shareholder_return の一部・Phase 2）

**提供する価値**: 取得枠・期間など、企業が株主へ資本をどう配分する決定をしたかを把握できる。

**受け入れる文書**: 自己株式取得の決議・結果を明示する開示。

**抽出項目とロジック**

| コンテンツ | 必須項目 | 抽出規則 |
| --- | --- | --- |
| 自己株式取得 | 決議日、取得上限株数、取得上限金額、取得期間 | 決議本文または表から、上限と期間が同じ決議を指すことを確認する |
| 自己株式取得結果 | 実施期間、取得株数、取得額 | 結果報告に明示された実績値を取得する |

**確定する企業ファクト**: 一自己株式取得の決議、または一取得結果。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜自己株式取得
<取得期間>に上限<株数>株・<金額>円の取得を決議
~~~

**投稿しない条件**: 株数・金額・期間のいずれかが欠ける場合。株主還元の良し悪しを表す評価を付ける場合。

## 資金調達・買収・売却・設備投資（capital_transaction・Phase 2）

**提供する価値**: 企業が資金をどのように調達し、何の資産・事業・設備へ配分する決定をしたかを把握できる。

**受け入れる文書**: 新株発行、社債・借入、M&A、事業・資産の取得または売却、設備投資の決議を明示する開示。
単なる検討開始や観測記事は対象にしない。

**抽出項目とロジック**

- transaction_type を、発行、借入、取得、売却、設備投資のいずれかへ文書種別と明示語から対応付ける。
- counterparty、target_asset_or_business、announced_at、effective_at、amount、funding_source を、根拠箇所に存在する場合だけ取得する。
- amount は通貨・単位を別項目として保持し、取得額・売却額・投資予定額・調達額を混同しない。
- 対象と金額が同じ取引を指すことを、同じ見出しまたは表の行で確認する。
- 目的は会社が明示した文だけを company_explanation として別ファクトにする。

**確定する企業ファクト**: 一つの資金調達、一つの取得・売却、一つの設備投資決定、または会社が明示した一つの目的。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜資本配分
<対象>の<取得・売却・設備投資・資金調達>を<時点>に決定
~~~

金額が検証できる場合だけ、二文目へ追加する。

~~~text
金額は<金額><通貨・単位>
~~~

**投稿しない条件**: 取引相手・対象・決定時点を特定できない場合。検討段階を決定済みと扱う場合。
会社が書いていない将来の利益効果を加える場合。

## 事業構造の変化（business_structure・Phase 3）

**提供する価値**: 企業がどの事業、製品・サービス、地域、顧客層を収益源とし、どの事業を開始・終了・
再編したかを把握できる。

**受け入れる文書**: セグメント変更、事業譲渡・取得、製品・サービス提供開始・終了、地域展開を正式に
公表する開示・公式発表。事業の説明だけで新たな変化がない定型的な会社紹介は自動投稿しない。

**抽出項目とロジック**

- structure_change_type を、新設、開始、終了、統合、分離、地域拡大、地域撤退の許可語から決める。
- business_unit、product_or_service、region、customer_scope、effective_at を、同じ根拠箇所から抽出する。
- 事業単位の名称は原文の名称を保持し、bot が新しい市場区分や顧客層を推定しない。
- 変更前後のセグメントを表が示す場合は、表の行名・列名を根拠として対応付ける。

**確定する企業ファクト**: 一つの事業・サービス・地域に対する一つの構造変化。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜事業
<時点>に<事業・製品・サービス>を<開始・終了・統合・分離>
~~~

地域が直接根拠としてある場合だけ、次を追加する。

~~~text
対象地域は<地域>
~~~

**投稿しない条件**: 対象事業と変化の行為が同じ根拠箇所で結び付かない場合。会社紹介の一般文を、
新規事業開始と解釈する場合。

## 戦略・数値目標・資源配分（strategy_target・Phase 3）

**提供する価値**: 経営が何を目指し、いつまでに何を達成する方針を示し、どこへ資源を配分すると
表明したかを把握できる。

**受け入れる文書**: 中期経営計画、事業計画、資本政策、投資方針など、会社が対象期間とともに公式に示す資料。
発言者・発表主体・対象期間が分からない記事は対象にしない。

**抽出項目とロジック**

- strategy_type を、数値目標、投資方針、撤退方針、製品戦略、地域戦略、資本方針へ対応付ける。
- owner を会社または明示された経営主体として取得する。
- target、deadline、target_value、unit、allocation_target を、同じ見出しまたは表行から抽出する。
- 数値目標は target_value と deadline がそろう場合だけ数値として投稿する。
- 資源配分は投資先・対象事業・金額または期間が明示される場合だけ確定する。

**確定する企業ファクト**: 一つの目標、一つの方針、一つの資源配分。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜戦略
<発表時点>に<対象>について<方針または目標>を公表
~~~

数値目標は次の形を使う。

~~~text
<期限>の<指標>目標は<数値><単位>
~~~

**投稿しない条件**: 会社が明示した目標と、bot が推定した目標を混同する場合。期限、対象、指標の
いずれかを欠く数値目標。過去の資料を新しい意思決定として再投稿する場合。

## 事業の実行・進捗（business_execution・Phase 3）

**提供する価値**: 企業が掲げた方針が、製品、契約、提携、設備、許認可、事業再編としてどこまで実行に
移ったかを把握できる。

**受け入れる文書**: 製品・サービスの提供開始、契約・提携の締結、設備の稼働・着工、許認可、事業再編の
完了・延期を明示する公式発表・開示。ContentSpec は、受け入れる execution_type と、対象となる
公式文書種別を明示しなければならない。

**抽出項目とロジック**

- execution_type を、開始、締結、稼働、着工、取得、完了、延期、終了の許可語に対応付ける。
- subject は製品・サービス・設備・契約・提携の名称、counterparty は相手方が明示される場合だけ取得する。
- occurred_at は発表日ではなく、開始日・稼働日・契約日などの実行時点が明記される場合に取得する。
- status が予定、開始、完了、延期のいずれかを表す原文を保持し、予定を完了として投稿しない。
- 契約金額・期間・地域は、同じ契約・製品・設備に対応する場合だけ追加する。

**確定する企業ファクト**: 一つの実行イベントまたは一つの進捗状態。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜実行
<対象>を<開始・締結・稼働・完了・延期>
~~~

相手方または実行時点が根拠にある場合だけ、二文目へ追加する。

~~~text
<相手方または実行時点>
~~~

**投稿しない条件**: 計画・検討・契約締結・提供開始・完了の状態を取り違える場合。相手方の推定、
契約効果の推定、発表日を実行日として扱う場合。

## リスク・制約（risk_constraint・Phase 3）

**提供する価値**: 企業が自ら開示した不確実性、損失、規制・供給・生産上の制約と、その対応を継続して
把握できる。

**受け入れる文書**: 業績予想の修正、減損・損失、訴訟・調査、事故・供給制約、規制対応、操業停止・再開を
明示する開示・公式発表。

**抽出項目とロジック**

- risk_type を、予想修正、損失、減損、訴訟、調査、供給制約、稼働制約、規制、事故へ対応付ける。
- affected_subject、occurred_at、affected_period、amount、company_response を、根拠箇所から個別に抽出する。
- 会社が原因や見通しを説明する場合は company_explanation として保存し、事実・金額・対応と別の company_fact にする。
- 業績予想の数値修正は[業績予想の修正](./forecast-revision.md)の表抽出を優先し、リスク投稿は数値以外の制約を扱う。

**確定する企業ファクト**: 一つのリスクまたは制約、一つの損失・減損、一つの会社対応。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜リスク
<時点>に<対象>で<制約・損失・調査・規制対応>を開示
~~~

金額または対象期間が根拠にある場合だけ、二文目へ追加する。

**投稿しない条件**: 憶測、報道、株価への影響をリスクとして加える場合。会社が原因を述べていないのに
因果を補う場合。解消済みの制約を現在のリスクとして再投稿する場合。

## ガバナンス・経営体制（governance_change・Phase 3）

**提供する価値**: 誰が、どの役割・組織で企業を運営し、どの統治上の決定が行われたかを把握できる。

**受け入れる文書**: 代表取締役・役員の異動、重要な組織再編、委員会・統治方針の決定を明示する
開示・公式発表。ContentSpec は、自動投稿の対象とする役職と組織の一覧を明示する。

**抽出項目とロジック**

- governance_type を、就任、退任、異動、組織新設、組織変更、方針決定へ対応付ける。
- person_or_organization、role_before、role_after、effective_at、decision_body を同じ根拠箇所から取得する。
- 異動では新しい役割と発効日を必須にし、推測した後任・権限範囲は使わない。
- 組織変更では、変更対象と変更行為を必須にする。
- 人事は、ContentSpec の対象役職に一致する場合だけ候補にする。

**確定する企業ファクト**: 一人の役員・代表者の一つの異動、一組織の一つの変更、一つの統治上の決定。

**投稿テンプレート**

~~~text
<企業名>（<コード>）｜ガバナンス
<発効日>付で<氏名または組織>を<就任・退任・変更>
~~~

**投稿しない条件**: 発効日、役割、変更対象のいずれかが欠ける場合。人事予定を確定事項として扱う場合。
軽微な定型的人事を重要な統治変更として投稿する場合。

## 増減要因・修正理由（Phase 2 で設計）

決算実績の増減要因、業績予想修正の理由など、会社が数値とともに明示した説明は価値が高いが、
文章抽出の機構が必要である。Phase 2 の text-rule 設計時に、数値ファクトと別の company_fact
（company_explanation）として扱う。数値から bot が要因を推測してはならない。

## 優先度と重複キー属性（凍結種別）

有効化時に種別ページへ移し、[共通契約](./common.md)の原則（company_id と ContentKind を追加、
版・ハッシュは含めない）で物理化する。

| 種別 | 初期 priority | 重複キーに必ず含める属性 |
| --- | ---: | --- |
| 自己株式取得（shareholder_return） | 90 | action_subtype、decision_date、acquisition_period、取得上限（株数・金額） |
| capital_transaction | 90 | transaction_type、target_asset_or_business、decision_date、effective_at |
| business_structure | 60 | structure_change_type、business_unit、region、effective_at |
| strategy_target | 60 | strategy_type、target、deadline、target_value |
| business_execution | 70 | execution_type、subject、counterparty、occurred_at、status |
| risk_constraint | 100 | risk_type、affected_subject、affected_period、occurred_at |
| governance_change | 50 | governance_type、person_or_organization、role_after、effective_at |
