# 決算実績（financial_result）

## 提供する価値

特定の会計期間に、企業の売上高・利益がいくらだったか、同じ条件で比較できる基準に対してどの程度
違うかを確認できる。数値を「業績改善」といった評価語へ置き換えない。

## 受け入れる文書

会計期間、連結範囲、単位、実績値を表で確認できる決算短信。TDnet の開示一覧の行から次をすべて
満たすものを論理文書種別 earnings_report として受け入れる。

1. 銘柄コードが対象銘柄集合に含まれる。
2. 表題が決算短信のパターンに一致する。初期パターン（正規表現は設定で持つ）:
   「〜年〜月期 決算短信」または「〜年〜月期 第〜四半期決算短信」を含み、会計基準の角括弧
   （〔日本基準〕〔IFRS〕〔米国基準〕）を含む。〔JMIS〕等その他の会計基準は初期パターンに含めず、
   原本保存のみとする（対象銘柄に該当会社を入れる場合は、要素体系をフィクスチャで確認してから
   ContentSpec の版上げで追加する）。
3. XBRL の添付がある。ない場合は原本（PDF）を保存し、分析対象にしない。

表題から取るメタデータ:

| 項目 | 取り方 |
| --- | --- |
| 会計期間の期末 | 表題の「〜年〜月期」から期末年月を得る。日は月末と仮定する。決算期変更等で XBRL コンテキストの期間と矛盾する場合は period_mismatch で抑止される |
| period_type | 表題に第 n 四半期があれば Qn 累計、なければ通期（FY） |
| accounting_standard | 角括弧内。XBRL 内の表記と食い違う場合は抑止 |
| is_correction | 表題に「訂正」を含むか |

REIT・インフラファンドの短信様式は対象外とする（対象銘柄集合に入れない前提。入っていた場合は
様式判定に失敗して抑止される側に倒す）。

## XBRL サマリーの読み取り

「要確認」と付けた項目は、実開示から採取するフィクスチャで確定してから実装に固定する
（→ [スコープと段階計画](../scope.md) M1）。

### パッケージ構成（要確認）

TDnet の XBRL は zip で配布され、サマリー（短信 1 ページ目の表に対応するインライン XBRL）と
添付資料に分かれる。読むのはサマリーだけとする。ファイル配置・命名はフィクスチャで確定する。

### 指標と要素の対応（要確認）

正規化 metric ごとに、taxonomy 要素の候補リストを ContentSpec の同義語設定として持つ。
候補リストの優先順で解決し、複数要素が同時に値を持つ場合は抑止する。

| metric（内部キー） | 日本基準の要素候補（例） | IFRS の要素候補（例） |
| --- | --- | --- |
| net_sales | NetSales、OperatingRevenues（サービス・運輸等）、OrdinaryRevenues（銀行）、InsurancePremiumsAndOthers（保険）など業種別 | TotalRevenues 系 |
| operating_profit | OperatingIncome | OperatingProfit 系 |
| profit_owners | ProfitAttributableToOwnersOfParent | 同名 IFRS 系 |
| eps（保存のみ） | NetIncomePerShare | BasicEarningsPerShare 系 |

重要な帰結: 売上高に相当する要素は業種で名称が異なる。**表示名は開示上の名称**（営業収益、経常収益など、
要素に対応する日本語ラベル）を使い、内部の重複キー・優先度判定は正規化 metric キーで行う。
銀行の経常収益を「売上高」と表示するような読み替えはしない。

候補リストにない要素しか存在しない短信は、その metric だけを抑止する。候補リストの追加は
ContentSpec の版上げで行う。

### コンテキストの選択（要確認）

サマリーの各数値は、期間・連結区分・実績予想区分の組（XBRL コンテキスト）に紐づく。
物理的なコンテキスト ID 命名はフィクスチャで確定する。論理規則:

| 論理軸 | 当期実績に使う値 | 前年同期（比較元）に使う値 |
| --- | --- | --- |
| 期間 | 当期の通期または Qn 累計 | 前期の同じ期間種別 |
| 連結区分 | 連結。連結が存在しない会社だけ個別 | 当期と同じ区分 |
| 実績・予想区分 | 実績 | 実績 |

- 前年同期、通期、四半期、累計、連結、個別を混在させない。四半期単独（3 か月）の値は扱わない。
- 連結と個別の両方がある場合、投稿対象は連結のみ。個別値は financial_fact として保存だけする。
- サマリーに含まれる通期予想の値（予想区分）は financial_fact として保存する。投稿はしない
  （予想の変更は[業績予想の修正](./forecast-revision.md)の開示だけを事実として扱う）。

### 値の正規化

- 生値（円）は、ix 要素の表示文字列に format・scale・sign 属性を適用して復元し、64bit 整数で保存する
  （unitRef は JPY であること）。小数を含む値（EPS など円銭）は文字列のまま保存し、計算に使わない。
- 表示値（display_value）は原文の表示文字列をそのまま使い、bot は数値を再フォーマットしない。
  これにより本文の数値は短信 1 ページ目の印字と常に一致する。フィクスチャで PDF の印字と突合して確認する。
- 表示単位は scale 属性から決める。scale 6 は百万円、scale 3 は千円。それ以外は抑止。
  表ヘッダの単位表記（百万円・千円）を取得できる場合は突合し、矛盾したら抑止する。
- decimals 属性は表示単位ではなく精度の宣言として扱う。復元した生値が decimals の示す精度と
  矛盾する場合は抑止する。
- 負値は sign 属性と原文の△表記から解釈する。nil・空・ハイフン表記の要素は「値なし」として扱う
  （エラーではない）。

### 差分の計算

- delta = 当期実績の生値 - 前年同期実績の生値。両者の metric・期間種別・連結区分・表示単位が
  一致する場合だけ計算する。
- change_rate は前年同期が 0 でなく、両値が同符号の場合だけ計算し、保存のみとする（本文に出さない）。
- 前年同期の値がサマリーにない場合（新規上場・決算期変更を含む）、baseline_type を none とし、
  比較なしのテンプレートで投稿する。
- 会社予想との差（baseline_type = company_forecast）は MVP では作らない。前年同期比だけとする。
  保存済みの予想 financial_fact との突合は Phase 2 以降で設計する。

## 確定する企業ファクト

一つの短信から、metric ごとに最大一つの company_fact を作る。

- 対象 metric は net_sales、operating_profit、profit_owners の 3 つ。eps ほかの抽出済み指標は
  financial_fact 保存のみとし、投稿対象の追加は ContentSpec の版上げで行う。
- 実績値と検証済みの比較差は一つの company_fact（一表示）にまとめ、baseline_type 属性で比較基準を
  区別する。実績値ファクトと差分ファクトを別々に作らない。
- 会社が数値とともに明示した増減要因は、文章抽出が必要なため MVP では扱わない（→ [将来種別（凍結）](./future.md)）。
  数値から bot が要因を推測してはならない。

属性の JSON 形（content_spec_version が型・必須性を検証する）:

~~~json
{
  "metric": "operating_profit",
  "metric_label": "営業利益",
  "fiscal_year_end": "2026-03-31",
  "period_type": "FY",
  "consolidation_scope": "consolidated",
  "accounting_standard": "jgaap",
  "actual_raw": 4532000000,
  "display_unit": "百万円",
  "display_value": "4,532",
  "baseline_type": "prior_year",
  "baseline_raw": 4020000000,
  "delta_raw": 512000000,
  "delta_sign": "+",
  "delta_display": "512",
  "correction_kind": null
}
~~~

根拠箇所（source_excerpt）は、当期実績セルと前年同期セルに対応する table 種別のレコードを参照する。
locator には要素名・コンテキスト ID・unitRef・scale・decimals を記録し、原本 XBRL 内で再特定できるようにする。

## 投稿テンプレート

~~~text
<企業名>（<コード>）｜決算
<期間表示>の<指標表示>は<実績表示><単位>。前年同期比<符号><差分表示><単位>
~~~

前年同期がない場合:

~~~text
<企業名>（<コード>）｜決算
<期間表示>の<指標表示>は<実績表示><単位>
~~~

| 表示 | 規則 |
| --- | --- |
| 企業名 | 銘柄マスターの名称。略称置換は初期は空（機構は ContentSpec の置換表として保持） |
| 期間表示 | 通期は「2026年3月期 通期」、四半期は「2026年3月期 第2四半期累計」 |
| 指標表示 | 開示上の名称（metric_label）。「親会社株主に帰属する当期純利益」も省略しない |
| 実績表示 | 原文の表示文字列をそのまま使う（△は - に正規化） |
| 差分表示 | 符号なしの絶対値。表示単位の整数、3 桁カンマ区切り。符号はテンプレートの符号欄だけに置き、値には含めない |
| 符号 | 差分が正なら +、負なら -、ゼロなら ±0（このとき差分表示は 0） |
| 訂正 | correction_kind があるとき一行目末尾に「（訂正）」を付ける |

長い企業名・指標名で最長ケースを組んでも重み付き長 280（→ [共通契約](./common.md)）に収まる見込みだが、
超えた場合は抑止する。

## 優先度と重複キー

- priority = 80 + 指標順（営業利益 3、売上高 2、親会社株主に帰属する当期純利益 1）。
  同一文書内では営業利益、売上高、親会社株主に帰属する当期純利益の順に採用される。
  通期短信では配当（priority 90）と合わせた文書上限 3 件により、親会社株主に帰属する当期純利益は
  通常採用されない（→ [配当](./dividend.md)のサブ上限。この選抜は意図した値付けである）。
- 投稿重複キー:

~~~text
company_id / financial_result / fiscal_year_end / period_type /
consolidation_scope / metric / fact_type=actual / baseline_type
~~~

値（実績額）はキーに含めない。同じ期・同じ指標の実績は一度だけ投稿する。訂正ファクトだけが
同一キーでも投稿できる（→ [共通契約](./common.md)の重複キーの原則）。

## 訂正

- 訂正短信は、disclosure_document の corrects_document_id で元文書（同一銘柄・同一期末・
  同一 period_type の直近 earnings_report）を参照する。元文書を特定できない場合は抑止する。
- 訂正文書から作った company_fact は correction_kind = source とし、値が元と同じ metric は投稿しない
  （数値が変わった metric だけ訂正投稿する）。
- 数値情報のない訂正（誤字訂正など、XBRL 再添付なし）は原本保存のみで投稿しない。
- bot 自身の誤解析を直す system 訂正は、開示側の訂正とは別の経路で扱う（→ [配信](../distribution.md)の重複と訂正）。

## 投稿しない条件（抑止理由コード）

[共通契約](./common.md)の共通抑止（鮮度・重複・上限など）に加えて:

| 理由コード | 条件 |
| --- | --- |
| no_xbrl | 短信表題だが XBRL 添付がない |
| metric_unresolved | 要素候補が 0 件、または複数要素が同時に値を持つ |
| unit_invalid | scale が 3・6 以外、unitRef が JPY 以外、表ヘッダの単位表記と矛盾、decimals の精度と生値が矛盾 |
| period_mismatch | 表題の期末・period_type と XBRL コンテキストの期間が矛盾する |
| standard_mismatch | 表題の会計基準と XBRL の要素体系が矛盾する |
| correction_unlinked | 訂正表題だが元文書を特定できない |

## フィクスチャ

実開示から採取し、期待値を人手で確定してから実装する。個別値の期待値は PDF の印字と一致させる。

| # | ケース | 確認すること |
| --- | --- | --- |
| 1 | 通期・連結・日本基準（前年同期あり） | 3 metric の company_fact、本文が印字と一致 |
| 2 | 第 2 四半期累計・連結 | period_type と期間表示 |
| 3 | IFRS 通期 | IFRS 要素候補での解決、指標表示名 |
| 4 | 個別のみの会社 | 個別へのフォールバック |
| 5 | 千円単位の会社 | display_unit 判定と割り切れ検証 |
| 6 | 銀行（経常収益系ラベル） | 開示名称の表示、net_sales キーへの正規化 |
| 7 | 前年同期なし | baseline_type = none、比較なしテンプレート |
| 8 | 訂正短信（数値変更あり） | corrects 参照、変更 metric のみ訂正投稿、重複キーのバイパス |
| 9 | 同一開示の再取得 | source_poll の取得観測は増えるが、source_record・分析・投稿は増えない |
| 10 | XBRL なし短信 | no_xbrl 抑止、原本保存のみ |

受入基準: フィクスチャ全通過、印字一致、リプレイ再現、「要確認」の全確定（→ [検証計画](../verification.md)）。
