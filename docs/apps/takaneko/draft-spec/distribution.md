# 配信（X 投稿）

## 方針

Takaneko は、検証済みの企業ファクトを、作者本人が管理する非公開（protected）X アカウントへ投稿する。
目的は作者本人が自分のタイムラインで確認することだけであり、第三者への配信、公開 Web サイト、
外部通知（メール・Slack 等）、商用提供は行わない。

- 投稿先は作者が管理する protected アカウント 1 つに固定する。
- 承認済みフォロワーを持たない運用を前提とする。フォロー申請は承認しない。
- protected アカウントでも「作者だけが必ず閲覧できる」ことの技術的な完全保証は別問題である。
  このため、フォロワーを承認しない運用に加えて、投稿前検証（下記）で protected 状態と
  フォロワー数を機械的に確認する。
- 投稿対象は、根拠・数値・時点を検証済みの企業ファクトだけとする（→ [コンテンツ仕様](./content/common.md)）。

企業ファクトと投稿記録は SQLite に保存し、localhost の lab または CLI でも確認できる
（→ [ローカル分析サイト](./local-site.md)）。

## 実行モード

実行モードは設定（app_mode）で管理し、dry_run と live の二値とする（→ [運用と設定](./operations.md)）。

| モード | 挙動 |
| --- | --- |
| dry_run | 候補生成・選抜・重複抑止・監査記録まで本番と同一に行い、外部送信は一切しない。post_attempt に status = dry_run で記録する |
| live | 投稿前検証を通過した場合だけ、選抜済み候補を protected アカウントへ投稿する |

- 初期モードは dry_run とする。live への切替は、[検証計画](./verification.md)の dry_run ゲートを
  通過した後に、運用者が設定版の追加で行う（コード変更では行わない）。
- live 設定は投稿先アカウント ID（数値）と月間予算上限を必須項目とする。欠けた live 設定は取り込みを拒否する。
- dry_run と live で候補生成・重複キー・選抜の挙動を変えない。差は外部送信の有無だけとする。

## 投稿前検証（フェイルクローズ）

live モードでは、投稿を行うサイクルの冒頭で、X API の自アカウント照会を使って次を確認する。

1. 認証ユーザーの数値 ID が、設定に固定した投稿先アカウント ID と一致する。
2. アカウントが protected である。
3. フォロワー数が 0 である。

いずれかを満たさない、または確認自体に失敗した場合は、そのサイクルの投稿を全て中止し、
全体キルスイッチを入れて operation_event に理由を記録する。解除は運用者の手動操作だけとする。
確認できないまま投稿してはならない（フェイルクローズ）。

## 投稿候補

company_fact は、次の条件をすべて満たす場合だけ投稿候補（post_candidate）にする。

1. 対象銘柄集合と有効な ContentSpec に一致する。
2. 根拠箇所、必須属性、品質状態（quality = postable）を検証できる。
3. 全体・情報源・ContentSpec のキルスイッチが解除されている。
4. 文書の訂正関係を確認できる。
5. 本文が種別ごとの固定テンプレートから生成され、重み付き長 280 以内である（→ [共通契約](./content/common.md)）。
6. 公開時刻から 24 時間以内である（文書鮮度。停止からの復帰で追補した古い開示は stale として抑止し、
   投稿しない。企業ファクト自体は保存され lab で閲覧できる）。

候補には、企業名・コード・種別・対象時点・数値と単位・根拠箇所への参照を保持する。

## 選抜と頻度

頻度・選抜の上限は配信ルール（設定）で管理する（→ [運用と設定](./operations.md)の初期設定値）。

- 一文書から投稿する候補は最大 3 件。priority と元文書内の出現順で選ぶ（→ [共通契約](./content/common.md)）。
- 銘柄・種別・アカウント単位の日次上限、15 分単位の上限、最小投稿間隔を守る。
- 一括選抜の時点で、approved・sending・unknown の候補・試行を頻度枠として予約し、
  送信直前に枠と重複キーを再判定する。承認済み未送信の候補が枠を超えて送信されてはならない。
- 上限から漏れた候補は抑止理由付きで保存し、翌日へ持ち越さない（鮮度切れとして扱う）。

## 送信と結果照合

post_attempt は候補ごとの送信試行を記録する。

| status | 意味 |
| --- | --- |
| dry_run | dry_run モードの記録。外部送信なし |
| sending | 送信要求を発行した直後 |
| published | X の投稿 ID を取得して成功を確認した |
| failed | 失敗を確認した（failure_class に分類を記録） |
| unknown | 成否を確認できていない（タイムアウト・応答欠落） |
| unknown_final | 照合を試みても成否を確定できず、打ち切った |

- 失敗した候補を自動で再送しない。ネットワーク起因と分類できた失敗のみ、次サイクルで 1 回だけ再試行してよい。
- unknown の照合は、読み取り API で自アカウントの直近投稿を取得し、本文または重複キーで突合できる場合だけ行う。
  照合できない場合は unknown_final とし、同じ重複キーの再送を恒久的に行わない（二重投稿よりも欠落を選ぶ）。
- 送信中・unknown の重複キーは新規送信の判定で「使用済み」として扱う。
- DB への保存に失敗している状態では投稿しない（→ [データモデル](./data-model.md)の無期限保存と容量）。

## 重複と訂正

同じ文書・根拠箇所・主題・ContentSpec 版から同じ company_fact を重複作成しない。重複キーの原則は
[共通契約](./content/common.md)に従う。

訂正は二系統を区別する。

| 系統 | 起点 | 扱い |
| --- | --- | --- |
| source 訂正 | 情報源の訂正開示 | 新しい文書・根拠箇所・企業ファクトとして保存し、correction_kind = source の候補は重複キーが同一でも投稿できる（訂正表示を含める） |
| system 訂正 | bot 自身の誤解析 | 運用者が CLI で対象の投稿（x_post_id）と正しい値の根拠を指定し、correction_kind = system の company_fact と訂正投稿を発行する。自動では発行しない |

いずれも過去の記録を上書き・削除しない。lab では訂正関係を表示する。

## 誤投稿時の対応

1. 全体キルスイッチを入れ、新規投稿を止める。
2. 誤った投稿は、作者が X 上で手動削除する。bot は投稿削除 API を持たない。
3. 値の誤りは system 訂正（上記）で訂正投稿できる。削除だけで済ませるか訂正投稿するかは運用者が判断する。
4. 原因と対応を operation_event に記録してから解除する。

## X の利用規約・自動化ルールへの適合

- アカウントに自動化ラベルを設定し、プロフィールに運営者（作者本人）を明示する。
- API の利用は投稿と自アカウント確認・結果照合に限る。live 切替前に Developer Console で
  従量課金の予算上限を設定し、月間予算（初期値 → [運用と設定](./operations.md)）を天井とする。
- 認証情報はローカルにだけ保存し、Git 管理しない（→ [運用と設定](./operations.md)）。

## スコープ外

- 公開アカウントでの投稿、フォロワーの承認
- 第三者へのデータ再配信、公開 Web サイト、外部通知（メール・Slack 等）
- 投稿削除 API、返信スレッド、引用・リポスト
- 投稿頻度・文字数を超えた選抜アルゴリズム（タイムライン最適化）

公開アカウントへの配信を将来検討する場合は、情報源のライセンス、配信先の利用条件、
本人による明示的な方針変更を先に満たす別プロジェクトとして扱う。
