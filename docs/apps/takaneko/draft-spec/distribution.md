# 配信

## 配信の前提

投稿先は本番 X アカウントだけとする。手動承認、検証用アカウント、投稿前の人手リリース工程は持たない。
投稿候補は自動で判定し、投稿可能なら直ちに X へ送る。実行モードが dry_run の間は、判定・記録まで行い
送信だけを止める（→ [運用と設定](./operations.md)）。

本文の生成規則（テンプレート限定、文字数上限、置換、文書あたり上限）と重複キーの構成は
[コンテンツ仕様: 共通契約](./content/common.md)を正本とする。本ページは、確定済みの company_fact を
安全に一投稿へ変換・送信する判定と執行を定める。

## 投稿候補の生成

distribution は一つの company_fact とその ContentSpec から、一つの post_candidate だけを作る。
共通契約の生成規則に従って本文を展開し、展開したテンプレート ID、表示項目、本文、重複キー、
対象集合・配信ルール版を post_candidate に保存する。生成段階で抑止された候補（文字数超過など）も
理由付きで保存する。

## 投稿候補の判定

候補は次の条件をすべて満たす場合だけ投稿可能（approved）になる。

1. 対象銘柄集合と company_fact 種別が配信ルールに合う。
2. company_fact の ContentSpec 版、根拠箇所、必須属性、品質状態が投稿可能である。
3. ルールが有効で、全体・情報源・分析・配信のキルスイッチが解除されている。
4. 文書と company_fact が鮮度条件を満たす（初期値 24 時間。→ [運用と設定](./operations.md)）。
5. 同じ企業ファクトを表す重複キーが、投稿済みまたは送信中ではない。
   訂正を表す company_fact だけは、訂正元と同じ重複キーでも投稿できる（→ 訂正）。
6. 文書あたり、銘柄ごと、分類ごと、アカウント全体の頻度上限を超えない。
7. 配信時間帯に含まれる。

判定はサイクルごとの一括選抜トランザクションで行う。頻度枠は、published だけでなく approved・
sending・unknown（unknown_final を含む）の候補も消費済みとして数え、承認の時点で枠を予約する。
これにより、承認済み未送信の候補が積み上がっても上限を超えて承認されない。送信の直前にも
キルスイッチ・鮮度・枠を再判定し、満たさなくなった候補は抑止へ落とす。

抑止した候補は、ContentSpec、抑止段階、理由コードとともに保存する。手動承認待ちという状態は存在しない。
抑止された候補は後から再判定して投稿しない（鮮度切れで自然に終了する）。

## 頻度と間隔の執行

頻度上限は、文書ごと、銘柄ごと、分類ごと、アカウント全体の四段階で持ち、どれかを超えた候補は投稿しない。
連続する投稿は最小間隔を空ける。初期値はすべて[運用と設定](./operations.md)の初期設定値に定める
（アカウント 1 暦日 10 件は X API 無料プランの月間上限に収める値。根拠は[スコープと段階計画](./scope.md)）。

一つの文書から複数の候補が出た場合も、前後の投稿を読まなければ理解できない連投や、タイムラインを
埋める文書要約にはしない（分割の上限と選抜は共通契約の生成規則）。

## X 投稿の状態と失敗分類

~~~text
candidate -> sending -> published
                   ├-> failed
                   ├-> unknown -> unknown_final
                   └-> suppressed
~~~

送信前に post_attempt を sending で保存してから X を呼び出す。結果の分類と処理:

| 分類 | 例 | 処理 |
| --- | --- | --- |
| published | 正常応答 | X 投稿 ID を保存 |
| failed（認証） | 401 / 403 | 再試行しない。該当する配信ルールを停止し operation_event に記録 |
| failed（レート） | 429 | 再試行しない。配信ルールを停止 |
| failed（本文不正） | 400 | 再試行しない。配信ルールを停止（テンプレート起因の疑い） |
| failed（一時・送信前確定） | 接続確立失敗など | 30 秒・5 分・30 分で最大 3 回再試行 |
| unknown | タイムアウト等、受理不明 | 自動再送しない。M0 で確認した照合手段が使える場合は、自アカウントの直近投稿の読み取り照合を 1 回だけ行う（unknown 発生時のみの従量費用）。確認できなければ 30 分後に unknown_final へ移し、重複キーを保持する |

不確定な候補を投稿し直して重複投稿を起こすことを避ける。プロセス再起動後も sending・unknown の状態を
保持し、自動再送しない。

## 訂正

訂正は correction_kind で二種類に分ける。いずれも本文の訂正表示を必須とし、重複判定では
correction_kind を持つファクトに限り訂正元と同じ重複キーでの投稿を許可する。訂正表示のない候補には
適用しない。

- **source**: 開示文書の訂正。訂正文書の分析から自動で生成する（種別ページの訂正規則に従う）。
- **system**: bot 自身の誤解析の修正。ContentSpec を直して再分析しただけでは、同じ重複キーに阻まれて
  自動投稿されない。運用者が CLI から対象の X 投稿 ID を指定して system 訂正を発行し、再分析済みの
  正しい company_fact から訂正投稿を生成する。発行・理由・対象は operation_event に記録する。

すでに投稿した内容への自動削除や、前の投稿に依存する返信投稿は初期機能に含めない。対象投稿と影響を
運用記録に残し、以後の新規投稿には訂正後のデータだけを使う。
