# NeteClay 技術検証計画

実装着手前に確認する技術項目。各項目は Android（ローカル dev build）と iOS（EAS ビルド + 実機）の
両方で確認する。Android を先に回し、通ったものを iOS で追試する。

## V-1: バックグラウンドでの音声再生の継続

- **確認**: 画面ロック後・他アプリ切り替え後も再生が継続すること。
- **Android**: メディア再生タイプの foreground service（`FOREGROUND_SERVICE` +
  `FOREGROUND_SERVICE_MEDIA_PLAYBACK`）。Android 13 以降は通知権限（`POST_NOTIFICATIONS`）も要る。
- **iOS**: オーディオセッション category `playback` + Background Modes `audio`。
- **成功条件**: 設定した総再生時間（仕様上の最大 75 分、余裕を見て 90 分で検証）を通してロック中も再生が途切れず、
  着信・他アプリ再生・Siri 起動での中断から自動復帰すること。
- **通知/ロック画面操作**: 停止は確実に動作させる。「もう一度あやす」は、直接実行できる場合は
  カスタムアクションで実行し、できない場合は通知タップでアプリへ戻して即操作できることを確認する。
- **通知権限別の確認**: 許可ありでは停止と「もう一度あやす」の操作導線を確認する。
  拒否時は再生継続とアプリ内操作だけを確認し、ロック画面/通知操作を成立条件にしない。
- **通知抑制**: アプリ単体で他アプリ通知を遮断しない。OS のおやすみモード/集中モード案内が
  Android / iOS で現実的に置けるか確認する。
- **注意**: バックグラウンド動作は Expo Go / dev client の Metro 接続では正しく検証できない。
  実ビルド（内部テスト配布 / TestFlight）で確認する。

## V-2: 2 トラック同時再生とクロスフェード

MVP 着手前の合否ゲートとする。ここが不成立なら、音楽エンジンの実装方式を見直してから実装に進む。

- **確認**: expo-audio でループ 2 トラックの同時再生・30〜60 秒の等パワークロスフェード・
  ギャップレスループが破綻なく行えること。
- **成功条件**: フェーズ移行時にクリックノイズ・音量の谷・ループの継ぎ目が知覚されないこと。
- **フォーマット**: WAV（無圧縮）を基準に、gapless メタデータ付き AAC を両 OS で比較し、
  品質と同梱サイズから書き出しフォーマットを決める（AAC / MP3 はエンコーダ遅延の無音が継ぎ目になりうる）。
- **不成立時**: ネイティブモジュール（AVAudioEngine / ExoPlayer）を用いる。

## V-3: バッテリー・発熱

- **確認**: V-1 の構成で仕様上の最大 75 分、余裕を見て 90 分再生したときの電池消費・発熱。
- **成功条件**: 寝室に置いて気にならない範囲。

## V-4: 音源パック容量と配布方式

- **確認**: 1 プリセットあたりの想定容量、無料同梱の容量上限、有料パックの配布方式を決める。
- **比較対象**: アプリ同梱、Android App Bundle、iOS On-Demand Resources。
- **測定対象**: メロディ系 1 種、環境音系 1 種、有料パック 1 つ分の合計サイズ。
- **MVP 方針**: まず無料 2 プリセットと有料 1 パックをアプリ同梱で成立させる。
  合計サイズが過大な場合だけ、OS 標準の分割配布へ切り替える。
- **判定**: MVP では無料同梱 2 プリセットを優先する。有料パックを含めた初回インストールサイズが過大なら、
  有料パックは App Bundle / On-Demand Resources など OS 標準の配布方式へ分ける。
- **成功条件**: 初回インストールサイズが過大にならず、サーバー不使用の前提と両立すること。

## V-5: タイムライングラフの操作性

- **確認**: 小さい画面・暗いテーマ・片手操作で「寝つくころ」「深くなるころ」の 2 点を誤解なく動かせること。
- **実装候補**: React Native View + SVG、または Canvas。MVP では保守しやすく、アクセシビリティ補助ボタンを付けやすい方式を優先する。
- **確認項目**:
  - ハンドル同士が近い場合でも選択できる。
  - ドラッグ中の分数表示が読みやすい。
  - ドラッグ中に値を即時反映しても操作が重くならない。
  - debounce 保存とドラッグ終了時 flush により、画面離脱後も最後の表示値が残る。
  - 1 分刻みの補助操作がスクリーンリーダーでも扱える。
  - 自動モードと手動モードの説明文が混乱しない。
  - 直近 100 件の手動モード記録を表示しても、グラフ操作が重くならない。
- **成功条件**: グラフを触らなくても既定値で始められ、触る場合も 2 点操作だと直感的に分かること。

## 進め方と成果物

1. V-1 を Android → iOS で確認する。
2. V-2 を並行して進める（フォアグラウンドで検証できる）。
3. V-3 を V-1 の構成で計測する。
4. V-4 で音源パックの容量と配布方式を決める。
5. V-5 でタイムライングラフの操作性を小さく試作する。

## MVP 着手ゲート

- V-1 が不成立なら MVP 実装に進まない。バックグラウンド再生が成立しない寝かしつけアプリは成立しない。
- V-2 が不成立なら、expo-audio 前提を捨ててネイティブ音声方式を検討する。
- V-4 で無料同梱 2 プリセットの容量が過大なら、MVP の音源数または配布方式を見直す。
- V-5 が不成立でも MVP は進められるが、その場合はタイムライングラフを簡素化し、補助ボタン中心の UI にする。

成果物: 各項目の結果（成功可否・計測値・ログ）、採用方式（expo-audio かネイティブモジュールか）、
バックグラウンド再生の設定一覧（実装時の app.json 仕様になる）。
