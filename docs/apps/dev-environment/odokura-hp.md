# odokura-hp の開発環境

このサイト（会社HP / Hatsu-go 紹介 / 規約・仕様メモ）自体の開発環境メモ。

## スタック

- Docusaurus 3.9.2 / React 19 / TypeScript 5.6
- Node.js 20 以上
- サイドバーは自動生成（`sidebars.ts` の `autogenerated`、フォルダ構成がそのまま反映される）

## 主なコマンド

```bash
npm start       # ローカル開発サーバー
npm run build   # 本番ビルド（draft: true のページは除外される）
npm run serve   # build 後の確認
npm run typecheck
```

## Claude / Codex の役割分担

`AGENTS.md` に正本がある。

- **Claude**: 設計 + Codex 向け実装プロンプトの執筆 + Codex の報告のレビュー。原則として feature code は書かない。
- **Codex**: 実装 + 検証。diff サマリ・検証コマンドの全出力・受け入れ条件ごとの自己チェックを報告する。
- ユーザーが「実装して」と直接 Claude に頼んだ場合は、Claude がコードを書いてよい。

詳細は下記の Skills セクションを参照。

## Skills (`.claude/skills/`)

- `codex-prompt`: Codex への実装プロンプト作成と、Codex の実装報告レビューに使う。プロンプトは「対象ファイル / 背景 / やってほしいこと / 制約 / 検証」の単一コードブロック形式にし、仕様に影響する変更は `docs/spec` を先に更新する仕様先行ルールを守る。
- `codex-git`: 実装フェーズ以降の git 操作を、Claude が直接実行せず Codex への一言指示に変える。たとえば「コミットして」「pushして」で起動する。
- `lazy-dev`: 実装・レビュー・Codex プロンプト設計の方針。YAGNI、標準機能や既存依存の活用、最小実装、不要な抽象・依存・ボイラープレートの排除を徹底する。

## git操作

実装フェーズ以降、Claude は `git status` / `diff` / `log` を含めすべての git コマンドを自分で実行しない。「コミットして」「pushして」のような一言を Codex に渡す。設計〜issue作成までの確認用 `status`/`diff`/`log` は Claude が直接見てよい。`gh`（issue/PR）は git操作ではないため対象外で、Claude が直接使う。詳細は Skills セクション参照。

## ドキュメント運用の注意

- 日本語ファイルは UTF-8 として明示的に読む（PowerShell: `Get-Content -Encoding UTF8`）。
- `docusaurus.config.ts` / `sidebars.ts` / navbar / footer / presets / routing / build・deploy 設定は、頼まれない限り content/CSS 作業中に変更しない。
- `src/pages/index.tsx` は公開中のトップページに合わせた特別なページ。通常の docs デザインに揃えない。
- `.md` は MDX なので `{ }` や バックティックを含む内容は必ずフェンス付きコードブロック（```）に入れる。インラインコードに入れるとビルドが落ちる。
- 仕様に影響する変更は `docs/spec`（`draft: true`、本番ビルドに出ない社内限定仕様）を先に更新し、実装をそれに合わせる（仕様先行）。
